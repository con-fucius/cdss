1. Concurrent incident search / list endpoint

Why it's impactful: 
Right now there is no way to list incidents at all — 'GET /incidents/{id}' retrieves one known incident by ID. 
Dispatchers and supervisors have no way to find an incident without already knowing its UUID. 
During a real call handoff or a shift change, operators need to search by status, time window, chief complaint, or assigned unit. 
The dashboard endpoints return aggregated counts but no individual incident records. This is a fundamental navigation gap.

What to note: 
Zero new tables. Zero new models. 
One new repository function ('list_incidents') using the existing 'Incident' table and '_incident_to_dict' serialiser. 
One new endpoint. No pagination library — a simple 'offset'/'limit' with validated bounds.

- [ ] Filter on 'Incident.pii_purged_at.is_(None)' to exclude purged records from search results — purged incidents are not usable

2. Protocol hot-reload without restart

Why it's impactful: 
Right now the protocol registry loads once at startup. 
Adding a new signed protocol, or substituting the PLACEHOLDER approval names with real ones, requires a full server restart. 
In a production environment running on a controlled server during a shift, a restart interrupts live calls — all in-progress incident state is fine (it's in Postgres), but the app is briefly unavailable. 
A hot-reload endpoint lets a supervisor trigger a registry reload without downtime. 
This directly unblocks the path to making the three proving protocols production-usable once medical director sign-off is obtained.

What to note: 
The registry already has 'load_all()'. The module-level 'registry' singleton already exists. 
This is one endpoint, one lock, two lines of logic. No file watching, no inotify, no background thread.

---
Checklist
'app/main.py'
- [ ] Add 'POST /admin/reload-protocols' — calls 'registry.load_all()' and 'field_registry.load_all()' in sequence, returns the result of 'list_active()' + 'list_rejected()' for both registries so the caller can confirm the reload result immediately
- [ ] Wrap both 'load_all()' calls in a try/except — a broken protocol file must not crash the reload or leave the registry in a partially-cleared state. If 'load_all()' raises unexpectedly (it currently doesn't, but defensively), catch, log, return 500 with the error
- [ ] Log '"Protocol registry reloaded by request"' at INFO level — this is an ops-relevant event
- [ ] No authentication added yet (out of scope), but add a comment: ' TODO: gate this endpoint on an admin key or network policy before production deployment'

Tests ('tests/test_protocol_reload.py')
- [ ] Test that calling 'load_all()' twice on the same registry produces the same result (idempotent reload)

Acceptance criteria
- [ ] 'POST /admin/reload-protocols' returns the new registry state (active + rejected for both dispatch and field)
- [ ] An incident in progress at the time of reload is unaffected — its 'dispatch_protocol_snapshot' (written at incident creation) is the source of truth, not the live registry
- [ ] 'list_rejected()' after reload accurately reflects the current file state, not a cached stale rejection from startup
- [ ] No background thread, no file watcher, no new dependency

3. Structured incident timeline endpoint

Why it's impactful: 
'GET /incidents/{id}/full' returns 6 separate arrays. 'GET /incidents/{id}/handoff' merges them into a document-oriented rendering. 
Neither gives a single chronologically-ordered timeline of everything that happened — dispatch answers, field actions, vitals records, and medications interleaved by timestamp. 
This is what a supervisor reconstructing a call or a medico-legal reviewer needs: one ordered sequence, not 6 arrays they have to manually merge and sort. 
It's also what 'handoff.text_rendering' builds manually by looping separate lists — that logic is already there, and a structured timeline endpoint just surfaces it directly.

What to note: 
No new tables. No new models. 
A single new repository function or 'main.py'-level aggregation merging the already-available sub-arrays by timestamp, returning a typed list of '{"timestamp", "event_type", "data"}' dicts. 
'handoff.py' already does essentially this (in rendering form) — this is making it available as structured data.

---
Checklist
'app/repositories.py'
- [ ] Add 'get_incident_timeline(incident_id: str) -> List[Dict]' — calls 'get_incident_full()' (which already parallelises its queries via 'asyncio.gather') and merges the 5 event streams into one chronologically sorted list
- [ ] Each row: '{"timestamp": str (ISO), "event_type": str, "source": str, "data": dict}' where 'event_type' is one of '"dispatch_answer"', '"field_action"', '"vitals"', '"medication"', '"guidance_lookup"'; 'source' is '"dispatch"' or '"field"' or '"system"'
- [ ] Sort by 'timestamp' ascending — ties broken by 'event_type' alphabetically (stable, deterministic)
- [ ] Rows with 'None' timestamp (should not happen in practice but be defensive) sorted last, not causing a crash
- [ ] Returns '[]' for an incident that exists but has no events yet — not 'None', not 404

'app/main.py'
- [ ] 'GET /incidents/{incident_id}/timeline' — 404 if incident not found; returns '{"incident_id": ..., "events": [...], "event_count": n}'
- [ ] No query params — always returns the full timeline. If pagination is ever needed, add it then, not now

Tests ('tests/test_timeline.py')
- [ ] Test merge of two events from different streams with different timestamps produces correct order
- [ ] Test that a 'None' timestamp row does not crash the sort
- [ ] Test that an empty incident returns '{"events": [], "event_count": 0}' — not an error

Acceptance criteria
- [ ] 'GET /incidents/{id}/timeline' returns a single chronologically ordered list spanning all event types
- [ ] An event's 'timestamp' matches the corresponding row in the source table (dispatch log, field log, etc.) — not generated at query time
- [ ] No new DB table, no new model, no new dependency
- [ ] 'get_incident_full()' is called once (not separately per event type)

4. NEWS2 trend alerting on vitals write

Why it's impactful: 
The system records NEWS2 at each vitals write and stores it. 
But nothing reacts to a deteriorating trend — a paramedic recording vitals every 5 minutes gets no signal that the patient's NEWS2 just crossed a threshold or jumped by 3 points since the last reading. 
This is the single highest-value clinical safety enhancement available without adding new infrastructure.
The data is already there (computed and stored in 'incident_vitals.news2_score'), the scoring logic already exists, and the write path ('add_vitals' in 'repositories.py') is the right place to compare the new score against the prior one.

What to note: 
No new table. No alerting service. No push notification. 
The output is a '"trend_alert"' field on the vitals-write response that the field UI already renders — the field console already receives the response from 'POST /incidents/{id}/vitals' and renders it. 
The alert is surfaced in the response, not via a background service. Callers already have to handle the response; this adds one field to it.

---
Checklist
'app/repositories.py' — 'add_vitals'
- [ ] After writing the new row, query the most recent prior vitals row for this incident (one 'SELECT' ordered by 'recorded_at DESC LIMIT 1' — scoped to the incident, not a full table scan)
- [ ] If a prior row exists and both 'prior.news2_score' and 'new.news2_score' are not 'None':
  - [ ] Compute 'delta = new_score - prior_score'
  - [ ] Classify: 'delta >= 3' → '"rapid_deterioration"', 'delta >= 1' → '"deteriorating"', 'delta == 0' → '"stable"', 'delta < 0' → '"improving"'
  - [ ] Also flag if new score crosses a risk-level boundary (low→medium or medium→high) regardless of delta magnitude
- [ ] Return the 'trend_alert' dict alongside the vitals row: '{"trend": str, "delta": int, "prior_news2": int|None, "new_news2": int, "crossed_risk_boundary": bool}'
- [ ] If no prior row, or either score is 'None': return '{"trend": "no_prior_data", "delta": None, ...}' — never omit the key entirely, field UI must be able to rely on its presence

'app/main.py' — 'add_incident_vitals'
- [ ] Pass through 'trend_alert' from the repository response into the HTTP response — it's already returned by 'add_vitals', just needs to not be discarded

Field UI ('field-ui/app.js' — 'loadVitalsHistory')
- [ ] On 'POST /incidents/{id}/vitals' response, if 'trend_alert.trend == "rapid_deterioration"' or 'crossed_risk_boundary == true', show a visible inline warning in the vitals tab above the history list: '"NEWS2 alert: [trend] — current [n], prior [n], delta [n]"'. Not a modal, not an 'alert()', not blocking — an inline '<div class="warning-banner">' that remains until the next vitals are submitted
- [ ] On 'trend == "improving"' or '"stable"': no banner, or clear any existing one

Tests ('tests/test_vitals_trend.py')
- [ ] Test: prior score 3, new score 6 → 'trend = "deteriorating"', 'delta = 3' (this is exactly '>=3' → 'rapid_deterioration' boundary)
- [ ] Test: prior score 5 (medium), new score 7 (high) → 'crossed_risk_boundary = True'
- [ ] Test: prior score 7, new score 4 → 'trend = "improving"'
- [ ] Test: no prior vitals → 'trend = "no_prior_data"', no error
- [ ] Test: prior score is 'None' (incomplete NEWS2 at last recording) → 'trend = "no_prior_data"', no crash

Acceptance criteria
- [ ] 'POST /incidents/{id}/vitals' response always includes 'trend_alert' key
- [ ] A 'rapid_deterioration' alert is visible in the field UI without a page reload
- [ ] No new DB table, no background thread, no push notification
- [ ] The trend comparison is one additional SELECT on the existing 'incident_vitals' table — no new query complexity

5. Incident export for medico-legal record

Why it's impactful: 
The handoff summary ('GET /incidents/{id}/handoff') is designed for a receiving facility. A medico-legal or audit record needs something different: 
a complete, immutable, timestamped export of every row associated with an incident (every question asked, every answer given, every vitals reading, every medication logged, every guidance lookup) formatted for printing or archiving, with explicit provenance labels (which system generated which field). 
Right now there is no endpoint that produces this. 
The 'GET /incidents/{id}/full' endpoint returns all the raw data, but it's JSON aimed at internal consumption, not a stable archival format. 'text_rendering' in the handoff is a start but omits raw timestamps and IDs needed for an audit trail.

What to note: 
No new tables. No new models. No PDF library (the output is plain text — a receiving system can PDF it if needed; a Python PDF dependency is over-engineering here). 
The data is already assembled by 'get_incident_full'. This is one new function in 'handoff.py' (a 'render_audit_text' function parallel to the existing '_render_text') and one new endpoint. 
The content is deterministic — two calls to the same incident return identical output — which is the property a medico-legal record needs.

---
Checklist

'app/handoff.py'
- [ ] Add 'render_audit_text(full: Dict) -> str' — takes the raw 'get_incident_full()' dict directly (not a 'HandoffSummary') so it has access to every row ID and raw timestamp
- [ ] Structure:
  - Header block: 'incident_id', 'created_at', 'status', 'chief_complaint', 'dispatch_protocol_id + version', 'dispatch_protocol_snapshot approval fields (approved_by, approved_date)'
  - Dispatch transcript block: every row from 'dispatch_log', formatted as '[{timestamp}] [{id}] Q: {question_text} | A: {answer} | protocol_v: {protocol_version}' — one row per line, '[BACKTRACK]' marker on backtrack rows
  - Guidance lookups block: every row from 'guidance_lookups', formatted as '[{timestamp}] [{id}] question_id: {question_id} | dispatcher: {dispatcher_id} | note: {result_summary}'
  - Vitals block: every row from 'vitals_history', all fields including raw component values, computed NEWS2, GCS
  - Medications block: every row from 'medications_given', including 'administered' flag, 'given_at', 'given_by'
  - Field actions block: every row from 'field_log', with 'step_id', 'action_type', 'recorded_by', full 'data' dict serialised as JSON on the same line
  - Footer: 'EXPORT GENERATED AT: {utc_now}' + 'INCIDENT DATA HASH: {sha256 of the serialised full dict}' — the hash lets a verifier confirm the export matches the stored data without needing to trust the export itself

'app/main.py'
- [ ] 'GET /incidents/{incident_id}/export' — returns 'PlainTextResponse' with 'Content-Disposition: attachment; filename="incident_{id}_audit.txt"' so browsers download it rather than rendering it
- [ ] 404 if incident not found
- [ ] No query params — always the full export

Tests ('tests/test_audit_export.py')
- [ ] Test that every row from 'dispatch_log' appears in the export text, identified by its 'id'
- [ ] Test that a backtrack row is marked '[BACKTRACK]'
- [ ] Test that the 'administered: false' medication flag appears in the export
- [ ] Test that the SHA256 hash in the footer changes when any field in the source data changes (compute hash of two slightly-different 'full' dicts and confirm the outputs differ)
- [ ] Test that the function is synchronous (no 'async def') and takes a plain dict — it must not make DB calls itself

Acceptance criteria
- [ ] 'GET /incidents/{id}/export' downloads a plain text file, not JSON
- [ ] Every row from every sub-table appears in the export with its database-assigned 'id' and raw timestamp
- [ ] The SHA256 hash at the footer is reproducible: calling the endpoint twice on the same incident returns the same hash
- [ ] 'administered: false' medications are labelled differently from 'administered: true' medications — a medico-legal reviewer must not have to infer this
- [ ] No new DB table, no new model, no PDF/docx dependency
