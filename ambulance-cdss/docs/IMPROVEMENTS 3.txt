1. Incident-level ETA tracking and overdue detection

What exists vs the gap: 
'dispatched_at' is stamped when a unit is assigned. 'on_scene_at' is stamped when the field unit calls 'POST /incidents/{id}/status' with 'on_scene'. 
The dispatch response returns 'eta_minutes' from the external service. 
But 'eta_minutes' is never persisted — it lives only in the 'DispatchResult' dataclass during the 'dispatch_unit' request and is gone after the response. 
There is no way to later ask "is this unit overdue" because the estimated arrival time was never stored. 
The dashboard endpoint returns counts of active incidents but no indication of which ones have been dispatched and are overdue for an on-scene report.

Why: 
an overdue unit — dispatched but not on-scene after 'eta_minutes' — is a patient-safety signal. 
Knowing about it in the dashboard is operationally necessary. 
The data needed (dispatch time, ETA) is already being received; it's just not persisted.

How: 
one new nullable column ('eta_minutes') on 'Incident', one migration, one line in 'set_assigned_unit' or a new 'set_dispatch_eta' call in the dispatch endpoint, and one computed field ('overdue: bool', 'minutes_since_dispatch: int') in '_incident_to_dict' and 'get_active_incidents'. No new table, no background job, no scheduler.

2. Protocol version mismatch detection

What exists vs the gap: 
when an incident is created, 'set_dispatch_protocol' snapshots 'protocol_id', 'version', 'approved_by', 'approved_date' into 'Incident.dispatch_protocol_snapshot'. 
When 'submit_incident_answer' runs, it fetches the live protocol from 'registry.get(incident["dispatch_protocol_id"])' — the current version, not the snapshot. 
If 'POST /admin/reload-protocols' has been called mid-incident (e.g. a corrected protocol was reloaded), the incident starts its call with v1.0.0 but answers are being processed against v1.0.1. 
The version stored in 'incident_dispatch_log' rows ('protocol_version') would then be v1.0.1 even though the incident's snapshot says v1.0.0. This is an undetected medico-legal inconsistency.

Why: 
a dispatcher mid-call should not be affected by a hot-reload. 
The snapshot was taken precisely to prevent this — but the runner never checks the live registry version against the snapshot version. 
The governance property the snapshot was meant to guarantee is not actually enforced.

How: 
a version comparison in 'submit_incident_answer' between 'incident["dispatch_protocol_snapshot"]["version"]' and 'protocol.version'. 
If they differ: return a structured warning in the response (not a hard error — the call must continue; stopping a live emergency call because of a version bump would be worse than the inconsistency). 
Log it. The dispatcher UI can surface the warning.

---
Checklist

If 'snapshot_version' is set and 'snapshot_version != protocol.version': log 'WARNING' with 'incident_id', 'snapshot_version', 'live_version' and set a 'version_mismatch_warning' flag
- [ ] Include '"version_mismatch": True, "snapshot_version": snapshot_version, "live_version": protocol.version' in the response body when the mismatch is detected — inside an optional '"warnings"' key so clients that don't check it are unaffected

- [ ] No new table, no new model, no new migration

3. Vitals completeness flag

What exists vs the gap: 
'add_vitals' accepts any combination of vitals fields — all optional. 
A paramedic under pressure might record only BP, or only a GCS, without the full set needed for NEWS2. 'compute_news2' raises 'ScoringError' for missing required fields, which 'add_vitals' catches silently with 'pass', storing 'news2_score=None'. 
The response back to the paramedic gives no indication of which specific fields are missing for a complete NEWS2 calculation. 
The field UI shows 'NEWS2 –' but the paramedic doesn't know whether it's because NEWS2 wasn't computed yet or because three specific fields are missing.

Why: 
the paramedic may not realise they need to enter 'consciousness' or 'temperature' to get a NEWS2 score. 
A missing NEWS2 on a deteriorating patient is a clinical information gap, not just a display gap. 
Surfacing which fields are missing makes the tool actively useful rather than silently incomplete.

How: 
'compute_news2' already raises 'ScoringError(message, missing_fields=[...])'. 
The 'missing_fields' list is already computed — it's just swallowed by the 'except ValueError: pass' in 'add_vitals'. 
Catching it, extracting 'missing_fields', and including them in the response as 'news2_missing_fields: List[str]' is a 3-line change to 'add_vitals' and a small addition to the field UI's vitals form handler.

4. Incident search by free-text chief complaint similarity

What exists vs the gap: 
'GET /incidents' filters by exact-match 'status', 'priority_code', 'assigned_unit_id', and datetime range. 
A dispatcher searching for a prior incident by typing "choking" or "not breathing" has no way to do it unless they already know the exact UUID or one of those exact filter values. 
The 'chief_complaint' column exists on every incident row and is a free-text string — but there is no endpoint to search it. 
This is the single most common lookup a dispatcher would do at shift change: "find me the cardiac arrest from two hours ago."

Why: 
without chief-complaint search, the incident list is only useful if you already know exactly what you're looking for. 
A 'ILIKE '%term%'' filter on 'chief_complaint' (case-insensitive substring match, supported natively by PostgreSQL) is a one-line SQLAlchemy addition to 'list_incidents' with a new query parameter.

How: 
no full-text index, no search engine, no embeddings. 
PostgreSQL's 'ILIKE' on a 'Text' column with a short-lived incident table is appropriate for the data volume (an ambulance service's incident table is not millions of rows; it's retention-purged at 30 days). 
If the table grows to a scale where 'ILIKE' is slow, add a 'pg_trgm' index then — don't add it speculatively now.

5. Pre-arrival instruction read-back confirmation log

What exists vs the gap: 
when a locked script reaches a terminal outcome, the dispatcher UI displays 'pre_arrival_instructions' as an ordered list and reads them to the caller. 
There is no record that this happened. The dispatch log records which answers were given but stops at the terminal outcome — the instructions themselves, and whether they were read and acknowledged, are not captured anywhere. 
The audit export lists 'pre_arrival_instructions' from the protocol as metadata, but there is no log row saying "dispatcher confirmed instructions were read at HH:MM."

Why: 
in a medico-legal context, whether pre-arrival instructions were given and confirmed matters. 
"The dispatcher read the CPR instructions" is a clinical and legal fact about the call that the system currently has no way to record. 
It is also useful operationally — a supervisor reviewing a call can see whether the terminal-outcome instructions were actually communicated.

How: 
this reuses 'incident_field_log' — the same append-only log already used for field actions. 
A confirmation is just a field log entry with 'action_type="pre_arrival_confirmation"', 'step_id="terminal_outcome"', and 'data={"outcome_id": "...", "instructions_read": True, "confirmed_by": "dispatcher_id"}'. 
No new table. One new endpoint or a thin layer on 'POST /incidents/{id}/field-log'. The dispatcher UI gets a "Confirm — instructions read to caller" button shown after the terminal outcome renders.

---
Every gap above is a real defect found in the code, not invented:
- 1 — 'eta_minutes' from the dispatch service response is discarded after the HTTP response. The overdue-detection capability the dashboard implicitly promises requires it to be persisted.
- 2 — 'submit_incident_answer' fetches the live protocol version, not the snapshotted one. The snapshot was added precisely to prevent mid-call version drift, but the check was never written.
- 3 — 'ScoringError.missing_fields' is computed and immediately swallowed. The paramedic sees 'NEWS2 –' and doesn't know which fields to enter to complete the score.
- 4 — 'list_incidents' has no text search. The most common dispatcher lookup ("find the cardiac arrest from earlier") is impossible without knowing the incident UUID.
- 5 — pre-arrival instructions are displayed and read but never confirmed or logged. This is a medico-legal gap in an append-only, audit-complete system that otherwise records everything.
