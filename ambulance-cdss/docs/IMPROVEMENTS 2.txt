1. Protocol reachability validation at load time

What exists vs. the gap: 
'_validate_branch_integrity' confirms every 'branch_map' target resolves to a real question or terminal outcome. 
What it does NOT check: whether every terminal outcome and every question is actually reachable from 'entry_question_id'. 
A question or outcome that can never be reached — because no branch points at it — is dead code in the protocol. 
Dead questions mean the author wrote guidance nobody will ever see. An unreachable terminal outcome means a priority code that can never fire. Neither is caught today. 
A protocol with a disconnected graph loads as "approved and active" despite the author defect.

Why: 
Once real approvals are in place and protocols are used in production, an unreachable outcome could mean a clinical pathway that looks authored but is silently inert. 
This is a governance defect that should be caught at load time, not discovered during a real call.

How: 
It's a graph traversal (BFS from 'entry_question_id') using only the data structures already loaded by '_parse_protocol'. 
No new models, no new tables, no new dependencies. One function added to 'registry.py', called at the end of '_parse_protocol' alongside the existing '_validate_branch_integrity'.

2. Incident close-out enforcement

What exists vs. the gap: 
'IncidentStatus' has 'HANDOFF_COMPLETE' and 'CLOSED' but nothing enforces that an incident must pass through 'HANDOFF_COMPLETE' before 'CLOSED', or that 'CLOSED' is a terminal state (once closed, it should not transition to 'ON_SCENE' or 'DISPATCHED'). The 'POST /incidents/{id}/status' endpoint accepts any valid 'IncidentStatus' value and sets it regardless of the current state. 
A dispatcher could accidentally close an incident before handoff, or reopen a closed incident. Neither is prevented or even warned.

Why: 
The retention purge ('purge_expired_incidents') uses 'closed_at' as its PII-purge trigger. 
An incorrectly-set 'closed_at' (e.g. set then reset to a different status) means either premature PII purge or indefinite PII retention. 
This is a data integrity and governance issue, not just a UX issue.

How: 
A transition table (dict mapping 'current_status -> allowed_next_statuses') is 10 lines. Checking it in 'update_incident_status' in 'repositories.py' is 5 lines. No new tables, no state machine library, no new models.

---
Checklist
'app/repositories.py'
- [ ] Define '_VALID_TRANSITIONS: Dict[IncidentStatus, set[IncidentStatus]]' mapping each status to the set of statuses it is permitted to transition to:
  - 'RECEIVED' → '{DISPATCHED, CLOSED}' (allow direct close if call cancelled)
  - 'DISPATCHED' → '{ON_SCENE, CLOSED}'
  - 'ON_SCENE' → '{TRANSPORTING, CLOSED}'
  - 'TRANSPORTING' → '{HANDOFF_COMPLETE, CLOSED}'
  - 'HANDOFF_COMPLETE' → '{CLOSED}'
  - 'CLOSED' → '{}' (terminal — no transitions permitted)
- [ ] In 'update_incident_status': before executing the UPDATE, fetch the current status of the incident and check 'new_status in _VALID_TRANSITIONS[current_status]'
- [ ] If the transition is not valid: raise a new 'InvalidStatusTransitionError(ValueError)' with 'current_status', 'requested_status', and 'allowed_statuses' fields — same pattern as 'OutOfScriptAnswerError' in 'runner.py' (structured error, not just a string)
- [ ] 'InvalidStatusTransitionError' lives in 'repositories.py' alongside the function that raises it

'app/main.py' — 'update_incident_status' endpoint

- [ ] The 422 body must be machine-readable (same pattern as 'out_of_script_answer' in 'submit_incident_answer') so a future dashboard can show the dispatcher which transitions are currently available

3. Weighted chief-complaint protocol selection with confidence score

What exists vs. the gap: 
'find_by_chief_complaint' returns the protocol whose longest matching trigger wins, single match. 
The caller (create incident endpoint) gets 'protocol_matched: True/False' but never knows how confident the match is, or whether multiple protocols had partial matches. 
For ambiguous presentations (e.g. "not breathing, bleeding from head") both the cardiac/respiratory protocol and the trauma protocol might partially match. 
The dispatcher gets no signal that this is an ambiguous case requiring manual verification.

Why: 
Wrong protocol selection at call creation is the highest-consequence error in this system. 
Knowing the selection was ambiguous — and surfacing that to the dispatcher — directly reduces the risk of a mismatched protocol running for several questions before anyone notices.

How: 
'find_by_chief_complaint' already does the work; extending it to return a confidence score (number of triggers matched / total triggers in the winning protocol, normalised 0–1) and the list of other partial matches is additive. 
Return type changes from 'Optional[DispatchProtocol]' to a new 'ProtocolMatchResult' dataclass. 
The endpoint already returns 'protocol_matched: True/False'; the confidence score and alternatives are added alongside it.

---
Checklist

- [ ] Add '@dataclass ProtocolMatchResult': 'protocol: DispatchProtocol', 'matched_triggers: List[str]', 'confidence: float' (0.0–1.0, calculated as 'len(matched_triggers) / len(protocol.chief_complaint_trigger)' — what fraction of the protocol's own trigger set matched the complaint), 'alternatives: List[ProtocolMatchResult]' (other protocols that had at least one trigger match, sorted by confidence descending)
- [ ] Rename 'find_by_chief_complaint' to 'match_by_chief_complaint(chief_complaint: str) -> Optional[ProtocolMatchResult]' — returns 'None' only when zero protocols had any trigger match
- [ ] Keep the longest-trigger-wins selection logic for choosing the primary; add tracking of all other protocols with ≥1 trigger match as 'alternatives'
- [ ] 'confidence' is always between 0.0 and 1.0 inclusive. If 'len(protocol.chief_complaint_trigger) == 0' (should never happen — registry rejects it — but defensive): return 0.0
- [ ] 'alternatives' excludes the winning protocol itself; is empty if no other protocol matched

- [ ] Add '"requires_manual_verification": True' to the response when 'confidence < 1.0' OR when 'alternatives' is non-empty — this is the actionable signal for the dispatcher UI, not just raw numbers

4. GCS trend alerting (parallel to NEWS2 trend, 4 in the previous set)

What exists vs. the gap: 
NEWS2 trending was added to 'add_vitals' — it computes '_compute_news2_trend' after every vitals write and returns 'trend_alert' on the response. 
GCS is also computed and stored at write time ('gcs_total' on 'IncidentVitals') but receives no trending. 
A dropping GCS is clinically as significant as a rising NEWS2 — it is the primary indicator of neurological deterioration — and it uses the same data and the same write path. 
The asymmetry between NEWS2 trend (surfaced, actionable) and GCS trend (computed, stored, silently ignored) is a gap.

Why: 
GCS dropping from 15 to 9 (a 6-point fall crossing the severe TBI threshold at ≤8) gets no clinical signal from this system right now. 
The data is in 'incident_vitals.gcs_total'. The write path is already the right place. The field UI already has the trend alert banner.

How: 
'_compute_news2_trend' is already a standalone function. 
A parallel '_compute_gcs_trend' follows the exact same pattern: fetch the prior row's 'gcs_total', compute delta, classify, return a dict with the same keys as 'trend_alert' but prefixed for GCS. 
The response from 'add_vitals' gains one new key ('gcs_trend_alert'). The field UI already has 'renderTrendAlert' — a second call with the GCS result reuses it.

---

5. Incident notes/free-text annotation endpoint

What exists vs. the gap: 
'incident_field_log' is used for everything the field unit produces — checklist steps, vitals (before the dedicated endpoint existed), medications (before their endpoint), and manual 'manual_entry' log rows (from 'field-log-form'). 
The dispatcher side has no equivalent: once a locked script reaches a terminal outcome, the dispatcher cannot add a free-text note to the incident record. 
There is no 'notes' field on 'Incident', no annotations table, and the 'GET /incidents' list endpoint already exposes 'incident.notes' (from the model) — but nothing writes to it. 
More critically: field log entries are write-only from the current endpoints — there is no structured 'note' retrieval outside 'get_incident_full'. 
A supervisor reviewing an active incident has to call '/full' and parse the entire assembled record to find a specific note.

Why: 
dispatcher-side annotation is operationally necessary. 
During a live call, a dispatcher receives radio updates from the field unit that are not captured by any structured endpoint — they go nowhere if the dispatcher has no way to append a note to the incident. 
These notes are part of the legally complete incident record, same as field log entries.

How: 
'Incident.notes' is already a 'Text' column in the model and migration (it's mapped in 'models.py' and '_incident_to_dict' already returns it). 
It just needs a 'PATCH /incidents/{id}/notes' endpoint that appends text to the existing 'notes' value with a timestamp prefix (so multiple appends build a chronological log, not overwrite each other). 
No new table. One field, one endpoint, one repository function.

---

---
Every item above is grounded in a real defect or gap found in the actual code, not a feature wish:
- 1 — the registry validates that branch targets exist but not that they're reachable. A protocol that silently hides unreachable outcomes is an undetected authoring error.
- 2 — 'update_incident_status' accepts 'CLOSED -> DISPATCHED' today. The retention policy uses 'closed_at' as a trigger. These two facts together are a PII governance risk.
- 3 — 'find_by_chief_complaint' returns a binary match/no-match with no confidence signal. The 'create_incident' response has no way to tell the dispatcher they should verify the protocol selection manually.
- 4 — GCS is computed and stored at every vitals write but never trended. The trend alert infrastructure exists (NEWS2), the data exists, the field UI banner exists. The gap is one function not yet written.
- 5 — 'Incident.notes' is a mapped, persisted column in the model with no write endpoint. Dispatcher-side annotation during a live call goes nowhere. The '_incident_to_dict' serialiser already returns it.
