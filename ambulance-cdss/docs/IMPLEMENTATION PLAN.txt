---
Critical design decisions I'm making and why

The three systems in the report must become three independently deployable services within one coherent ecosystem, not merged code. 
The ambulance-cdss remains the authoritative incident record and governance-gated decision engine. 
The Triage Ranker becomes its own service we build from scratch, informed by but not copying the NLP approach in the report. 
The Facility Mapper becomes its own service we build from scratch with the BallTree/geopy approach from the report but with a proper database, a versioned data schema, and a live-update path.

The RAG API (System 2) from the report is not a service we build independently. 
Its capability belongs inside the ambulance-cdss as Mode 2 guidance content, authored offline using RAG tooling, not as a runtime call. 
A runtime LLM call in the dispatch path is a patient safety risk — 2–10 seconds of generation latency, hallucination risk, and hard failure mode when the LLM is down. 
What the RAG API produces (protocol-grounded action steps) becomes the content of 'guidance_note' fields in protocol JSON files.
The authoring tooling for that is a separate offline workflow, not a production service.

System 3's capture layer (the portal form listener and HUD) is the field-facing UI surface.
We are not building an NHS portal injection — we are building a dispatcher console and a field console, which we already have as static HTML apps.
System 3's value is in its data flow design: the structured payload it builds, the real-time event coordination between form state and the clinical intelligence layer, and the report generation pattern. 
These inform how we harden and extend our existing dispatcher and field UIs.

The facility mapper is the most immediately valuable new service because 'FacilityRegistryClient' currently returns empty for every call. 
We are building it from scratch with real Kenya health facility data, proper geodata, and a versioned data store — not JSON files baked into a container.

## Implementation Plan

'''
Repository structure (each service is a first-class Python project)
D:\Projects\CDSS\
├── ambulance-cdss\          ← Authoritative incident record + dispatch engine (existing)
├── triage-ranker\           ← New: NLP clinical enrichment service
├── facility-mapper\         ← New: Geospatial facility routing service
└── shared\                  ← Shared Pydantic schemas consumed by all three
'''

Each service: independently runnable, independently testable, independently deployable. They communicate only via HTTP with documented contracts. No shared database. No shared in-process imports across service boundaries.

### Phase 0 — Shared contract schemas (prerequisite, ~1 day)

Why first: three services sharing a language requires defining that language before any service is built. Without this, each service invents its own field names and the integration layer becomes a translation mess.

'shared/contracts/'

- [ ] 'triage.py' — 'TriageRequest', 'TriageResponse', 'DiagnosisRankItem', 'ExtractedKeyword', 'TriageMetadata' as Pydantic v2 models. Exactly matching what 'ambulance-cdss''s 'TriageRankerClient' will send and receive.
- [ ] 'facility.py' — 'FacilitySearchRequest', 'FacilityResult', 'FacilitySearchResponse' as Pydantic v2 models. Exactly matching what 'ambulance-cdss''s 'FacilityRegistryClient' will send and receive.
- [ ] 'incident_capture.py' — 'CapturePayload', 'PatientInfo', 'IncidentInfo' as Pydantic v2 models. This is the structured payload the dispatcher UI sends to 'POST /incidents/from-capture' — the enriched incident creation endpoint.
- [ ] '__init__.py' per module, 'pyproject.toml' marking this as an installable package ('ambulance-cdss-contracts').
- [ ] Each service installs 'ambulance-cdss-contracts' as a local dependency — 'uv pip install -e ../../shared'.

Constraints:
- No logic in 'shared/' — only schemas. If a function appears here, it belongs in a service.
- Every field has a docstring explaining its clinical meaning. These schemas are read by developers who may not have a clinical background.
- Breaking changes to shared schemas require updating all three services simultaneously — tracked as a single commit, not three separate ones.

Acceptance:
- [ ] 'from ambulance_cdss_contracts.triage import TriageRequest' works from all three service directories
- [ ] All fields have type annotations and docstrings
- [ ] 'uv run python -c "from ambulance_cdss_contracts import triage, facility, incident_capture"' runs without error from any service directory

### Phase 1 — Facility Mapper service (highest immediate impact, ~3–4 days)

Why now: 'FacilityRegistryClient' returns empty on every call. This is the fastest path to a working end-to-end incident → facility routing flow. It is also a standalone service with no dependencies on the Triage Ranker, so it can be built, tested, and deployed independently.

Architecture:

'''
facility-mapper/
├── app/
│   ├── main.py          ← FastAPI app, endpoints
│   ├── config.py        ← env config
│   ├── data.py          ← facility data loading, validation, BallTree build
│   ├── geocoding.py     ← geocoding abstraction (Nominatim + fallback)
│   ├── matcher.py       ← BallTree KNN search, level filtering, ETA calculation
│   ├── models.py        ← SQLAlchemy models (Postgres, not JSON files)
│   └── schemas.py       ← local Pydantic schemas (re-exports shared contracts)
├── data/
│   └── README.md        ← data currency policy, update procedure
├── tests/
├── alembic/
├── pyproject.toml
└── .env.example
'''

Data storage decision: The report uses a baked-in JSON file. We use PostgreSQL with a 'facilities' table. Reason: a JSON file cannot be updated without redeploying the container. In a country where hospitals open, close, lose ICU capacity, or change level classifications, quarterly data refreshes are a patient safety requirement. A database supports versioned upserts, tracks 'data_as_of', and allows level/service filter queries without loading 10,000 records into memory on every request.

'facilities' table:

'''sql
facility_id       TEXT PRIMARY KEY,
name              TEXT NOT NULL,
county            TEXT,
level             INTEGER NOT NULL,    -- 1–6, Kenya KEPH levels
lat               DOUBLE PRECISION NOT NULL,
lon               DOUBLE PRECISION NOT NULL,
phone             TEXT,
services          TEXT[],              -- ["surgery", "blood_bank", "icu", "ct_scan"]
is_active         BOOLEAN NOT NULL DEFAULT TRUE,
data_source       TEXT NOT NULL,       -- e.g. "KMHFL_2024_02"
last_verified_at  TIMESTAMPTZ,
created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
'''

Tasks:

- [ ] 1.1 Project scaffold — 'pyproject.toml', '.env.example', 'alembic.ini', 'uv' venv, 'ruff' config identical to ambulance-cdss conventions
- [ ] 1.2 Data model and migration — 'facilities' table, 'data_imports' audit table (tracks when data was loaded, from what source, how many records), Alembic migration
- [ ] 1.3 Data loader — 'scripts/load_facilities.py': reads source data (Kenya KMHFL or equivalent), validates each row, upserts into 'facilities', writes a 'data_imports' row. CLI: 'uv run python -m scripts.load_facilities --source path/to/data.csv'. Must be idempotent — running twice produces no duplicates.
- [ ] 1.4 BallTree matcher — 'matcher.py': loads active facilities from DB at startup (cached in-process, invalidated by config TTL or explicit '/admin/reload-facilities' call). BallTree built from lat/lon coordinates using Haversine metric. 'find_nearest(lat, lon, level_min, required_services, radius_km, max_results)' → 'List[FacilityResult]'. Level filter applied post-KNN (filter, then re-rank by distance). ETA: '(distance_km / 60.0) * 60' minutes at 60 km/h — configurable speed via env var.
- [ ] 1.5 Geocoding — 'geocoding.py': 'geocode(location_text) -> Optional[tuple[float, float]]'. Uses Nominatim (free, no API key). 5s timeout. Returns 'None' on failure — never raises. Result cached in-process for 1 hour (location strings repeat constantly in a dispatch centre). This is the bridge between System 3's text-location approach and the lat/lon approach in 'FacilityRegistryClient'.
- [ ] 1.6 Endpoints — 'main.py':
  - 'GET /health' — '{"status": "ok", "facility_count": n, "data_as_of": "KMHFL_2024_02", "last_loaded_at": "..."}'
  - 'GET /ready' — 503 until BallTree is built
  - 'GET /facilities/nearest' — query params: 'lat', 'lon', 'radius_km', 'level_min', 'required_services', 'max_results' (default 3, max 10). Returns 'FacilitySearchResponse' from shared contracts.
  - 'POST /facilities/nearest-by-location' — body: '{"location": "Kiambu"}'. Geocodes, then calls same logic as above. This is the bridge endpoint for callers that have text location, not coordinates.
  - 'GET /facilities/{facility_id}' — single facility detail
  - 'POST /admin/reload-facilities' — clears in-process cache, rebuilds BallTree from DB. Requires 'X-Admin-Key'.
  - 'GET /data-currency' — returns 'data_imports' history for ops transparency.
- [ ] 1.7 Update 'ambulance-cdss' 'FacilityRegistryClient' — change '_build_nearest_request' and '_parse_nearest_response' to match 'FacilitySearchResponse' from shared contracts. Add 'POST /facilities/nearest-by-location' as a second search method when caller has text location. No other changes to 'ambulance-cdss'.
- [ ] 1.8 Tests — 'tests/test_matcher.py': BallTree returns closest facility; level filter excludes Level 3 when 'level_min=4'; empty result when no facilities within radius; ETA formula correct. 'tests/test_geocoding.py': cache hit returns same result without second HTTP call; timeout returns 'None' without raising. 'tests/test_data_loader.py': idempotent load produces same count on second run.

Constraints:
- No JSON file data storage. Postgres only.
- BallTree is built once at startup and on explicit reload — not per request.
- Geocoding is always cached. Repeat identical location strings cost zero external calls.
- 'data_as_of' field is always included in responses. A dispatcher UI must be able to display "Facility data as of [date]".
- ETA formula and ambulance speed are documented in code and configurable. They are not magic numbers.

Acceptance:
- [ ] 'GET /facilities/nearest?lat=-1.2921&lon=36.8219&level_min=4' returns ≤3 facilities with distance and ETA
- [ ] 'POST /facilities/nearest-by-location' with '{"location": "Kiambu"}' geocodes and returns facilities
- [ ] 'GET /health' includes 'data_as_of' field
- [ ] 'ambulance-cdss''s 'POST /incidents/{id}/route-facility' returns real facilities when Facility Mapper is running
- [ ] All tests pass without network (BallTree test uses seeded DB data, geocoding test mocks Nominatim)

### Phase 2 — Triage Ranker service (~4–5 days)

Why now: once the Facility Mapper is running, the Triage Ranker is the second highest-impact new service. It enriches every incident with structured clinical intelligence, improves protocol matching for non-English chief complaints, and feeds severity scoring into the facility routing decision (a P1 cardiac arrest should prefer a facility with a cath lab, not just the nearest Level 4 hospital).

Architecture:

'''
triage-ranker/
├── app/
│   ├── main.py          ← FastAPI app, endpoints
│   ├── config.py
│   ├── pipeline/
│   │   ├── __init__.py
│   │   ├── extractor.py      ← Stage 1: NLP entity extraction
│   │   ├── resolver.py       ← Stage 2: UMLS resolution with 4-layer cache
│   │   └── ranker.py         ← Stage 3: composite severity scoring
│   ├── cache/
│   │   ├── memory.py         ← L1: TTLCache (in-process)
│   │   └── persistent.py     ← L2: SQLite (persistent volume)
│   ├── rules/
│   │   └── clinical_rules.yaml  ← term → condition mappings, severity weights
│   └── schemas.py           ← re-exports from shared contracts
├── tests/
├── pyproject.toml
└── .env.example
'''

Key design decisions, grounded in the constraints of a country like Kenya:

Stage 1 (NLP extraction) must work offline. spaCy with 'en_core_web_md' runs offline once the model is downloaded. The model must be baked into the container — not downloaded at runtime. Kenya's network connectivity cannot be assumed during a live call.

Stage 2 (UMLS resolution) must degrade gracefully. If UMLS API is unreachable, Stage 1 output still reaches Stage 3. A triage level is still produced. This is the report's degradation strategy and it is correct.

Stage 3 (ranking) must work in Swahili and English. The 'clinical_rules.yaml' must include Swahili terms for the most common presentations (mshtuko = shock, maumivu ya kifua = chest pain, kizunguzungu = dizziness, kushindwa kupumua = difficulty breathing). The NLP model will not extract Swahili entities — the rules file compensates with keyword matching after NLP runs. This is not elegant but it is practical and correct for Kenya.

The UMLS API key is optional. Without it, Stage 2 runs from cache/fallback dict only. The service still produces useful output — just without canonical SNOMED/ICD-10 codes on cache misses.

Tasks:

- [ ] 2.1 Project scaffold — identical conventions to facility-mapper and ambulance-cdss
- [ ] 2.2 'clinical_rules.yaml' — English and Swahili term mappings to clinical conditions with severity weights. Categories: 'RESPIRATORY', 'CARDIOVASCULAR', 'NEUROLOGICAL', 'TRAUMA', 'OBSTETRIC', 'PAEDIATRIC'. Each entry: 'term', 'synonyms' (including Swahili), 'category', 'severity_weight', 'icd10_prefix', 'snomed_hint'. Minimum 50 terms covering the top presentations from Kenya emergency data. This file is the medical content — it must be reviewed by a clinician, not just a developer.
- [ ] 2.3 Stage 1 — NLP extractor — 'extractor.py': loads spaCy 'en_core_web_md' model (model path configurable, must not download at runtime). Tokenizes 'incident_desc'. Applies 'clinical_rules.yaml' term matching (case-insensitive, handles common misspellings via regex alternatives in rules file). Extracts 'acvpu', vital sign mentions, negation markers. Returns 'List[ExtractedKeyword]'.
- [ ] 2.4 Stage 2 — UMLS resolver — 'resolver.py' with four-layer cache: L1 in-process 'TTLCache' (1000 entries, 1 hour TTL using 'cachetools'), L2 SQLite persistent ('cache/umls_cache.db', survives restarts), L3 UMLS REST API (only called when L1+L2 miss, 3s timeout), L4 fallback dict derived from 'clinical_rules.yaml' 'icd10_prefix'/'snomed_hint' fields. On L3 success: write to L2 and L1. On L3 failure: use L4, set 'degraded_mode=True'. Never blocks Stage 3.
- [ ] 2.5 Stage 3 — ranker — 'ranker.py': composite score = 'w_rule' (rule match weight from 'clinical_rules.yaml') + 'w_semantic' (spaCy similarity if UMLS enriched) + 'w_modifier' (severity modifier from ConText: "severe", "critical", "acute") + 'w_scoring_system' (GCS component score if 'gcs_score' provided, Shock Index if 'sbp'+'hr' provided). Maps composite score to 'triage_level' (P1–P4) and 'esi_level' (1–5). Returns ranked 'List[DiagnosisRankItem]'.
- [ ] 2.6 Endpoints — 'main.py':
  - 'POST /triage' — main endpoint. Validates 'TriageRequest', runs all three stages, returns 'TriageResponse'. Max input length 5000 chars (validated by Pydantic). Processing time target: < 500ms on cache hit, < 5s on UMLS API call.
  - 'GET /health' — includes UMLS reachability (non-blocking check), cache stats, model loaded status.
  - 'GET /ready' — 503 until spaCy model is loaded.
  - 'POST /admin/rules/reload' — reloads 'clinical_rules.yaml' from disk without restart. Requires 'X-Admin-Key'.
  - 'DELETE /admin/cache' — purges L1 and L2 caches. Requires 'X-Admin-Key'.
- [ ] 2.7 Update 'ambulance-cdss' 'TriageRankerClient' — new 'app/external/triage_ranker.py' following exact same pattern as 'FacilityRegistryClient'. 'enrich(incident_desc, gcs_score, acvpu, sbp, hr) -> Optional[TriageEnrichment]'. Returns 'None' on any failure, never raises. Called as 'asyncio.create_task()' in 'create_incident' — non-blocking.
- [ ] 2.8 Migration '0006_triage_enrichment' — adds 'triage_enrichment JSONB' column to 'incidents' table. Nullable. Written asynchronously when the background task completes.
- [ ] 2.9 Update 'create_incident' in 'ambulance-cdss' — fire triage enrichment as background task. When it resolves, write result to 'incident.triage_enrichment'. Include enrichment in '/full', '/handoff', and '/export' responses. If enrichment is not yet resolved when '/handoff' is called, include '"triage_enrichment": null' rather than waiting.
- [ ] 2.10 Update 'match_by_chief_complaint' — when 'confidence < 0.8' or 'protocol_matched: false', use 'TriageEnrichment.triage_level' and 'top_diagnosis' as secondary matching signals if enrichment is already resolved (synchronous check on the 'triage_enrichment' column). Do not wait for enrichment — this is opportunistic, not required.
- [ ] 2.11 Tests — 'tests/test_extractor.py': "not breathing, unconscious" extracts NEUROLOGICAL and RESPIRATORY keywords. Swahili term "kushindwa kupumua" extracts RESPIRATORY. Negation "denies chest pain" does not extract chest pain as active. 'tests/test_ranker.py': GCS=3 produces P1 triage level. Shock Index > 1.0 (HR 120, SBP 90) produces 'severity_level=critical'. 'tests/test_resolver.py': L1 cache hit returns without calling SQLite. L2 hit does not call UMLS API. UMLS API timeout still produces a result via L4 fallback.

Constraints:
- spaCy model is baked into the container. 'python -m spacy download en_core_web_md' runs in Dockerfile, not at startup.
- 'clinical_rules.yaml' is a read-only volume in production. Changes require a reload call, not a restart.
- UMLS API key absence is handled at config level, not scattered through pipeline code. 'resolver.py' checks '_umls_configured()' once and skips L3 entirely if false.
- PHI handling: 'incident_desc' is never logged. The 'POST /triage' handler logs only 'request_id' and timing. This is non-negotiable.
- Stage 3 never returns zero results. If NLP finds nothing, the fallback is a single 'DiagnosisRankItem' with 'canonical_name="Undifferentiated Emergency"', 'triage_level="P2"', and 'degraded_mode=True'.

Acceptance:
- [ ] 'POST /triage' with '{"incident_desc": "patient collapsed, not breathing, no pulse"}' returns 'triage_level: "P1"' and at least one 'DiagnosisRankItem'
- [ ] 'POST /triage' with '{"incident_desc": "kushindwa kupumua"}' (Swahili: difficulty breathing) returns a RESPIRATORY-category result
- [ ] 'POST /triage' when UMLS API is down (mocked) still returns a result with 'degraded_mode: true'
- [ ] 'POST /triage' logs no patient text in any log output (verified by test that captures log output)
- [ ] 'ambulance-cdss''s 'create_incident' returns immediately without waiting for triage enrichment
- [ ] Integration: 'GET /incidents/{id}/full' for an incident created > 2 seconds ago includes 'triage_enrichment' with a non-null value when the Triage Ranker is running

### Phase 3 — Missing referenced protocols and governance (not optional, ~2 days)

Why now, not later: the cardiac arrest protocol routes to 'outcome_route_to_other_protocol' for "unresponsive but breathing" and "has pulse but abnormal breathing." These are live gaps. A dispatcher running the cardiac arrest protocol today can reach a terminal outcome that tells them "reassess and select another protocol" — and there is nothing to select.

Tasks:

- [ ] 3.1 'unresponsive_breathing_v1' dispatch protocol — entry: Is the patient breathing normally? Branches: breathing normally → recovery position instructions → P2_UNCONSCIOUS_BREATHING. Abnormal breathing → Is there a bystander who can maintain airway position? → P1_AIRWAY_COMPROMISE. Full branch coverage, all reachability validated.
- [ ] 3.2 'respiratory_distress_v1' dispatch protocol — entry: Can the patient speak in full sentences? Branches: yes → position of comfort, supplemental oxygen if available, monitoring → P2_RESPIRATORY_DISTRESS. No → Is there audible stridor/wheeze? → branches to P1_RESPIRATORY_FAILURE or P1_AIRWAY_COMPROMISE.
- [ ] 3.3 'obstetric_emergency_v1' dispatch protocol — entry: Is the patient pregnant or has she given birth in the last 6 weeks? Branches: active bleeding, seizure (eclampsia), cord prolapse. P1_OBSTETRIC_EMERGENCY throughout. Kenya's maternal mortality context makes this a required protocol, not optional.
- [ ] 3.4 'paediatric_respiratory_v1' dispatch protocol — common and high-stakes presentation in Kenya. Entry: How old is the child? Branches by age group (< 1 year, 1–5 years, > 5 years) with age-appropriate reference ranges.
- [ ] 3.5 'trauma_moi_v1' dispatch protocol — mechanism-of-injury triage. Not duplicating the major trauma MVA protocol but covering falls from height, assault, industrial accidents, burns. Kenya's road traffic mortality context.
- [ ] For each protocol: validated against reachability check, 'entry_question_id' confirmed reachable, no dangling terminal outcome references, 'approved_by'/'approved_date' PLACEHOLDER pending clinical sign-off (same governance posture as existing three), 'chief_complaint_trigger' includes both English and Swahili terms.

Constraints:
- No protocol references another protocol as a 'terminal_outcome' target without that protocol existing in the registry. The 'outcome_route_to_other_protocol' terminal outcome is a temporary placeholder — once the target protocols exist, it should be retired and each routing outcome replaced with a specific one.
- Every 'guidance_note' on gated questions is reviewed for clinical accuracy before the protocol file is committed.

Acceptance:
- [ ] 'GET /protocols' returns 8 active protocols (once signed off) with no rejections
- [ ] No terminal outcome in any protocol routes to a non-existent protocol
- [ ] Reachability validation passes for all 8 protocols
- [ ] Swahili chief complaint triggers present on at least the 3 highest-volume protocols (cardiac arrest, trauma, respiratory)

### Phase 4 — Enriched incident creation endpoint and System 3 data flow (~2 days)

Why now: once the Triage Ranker and Facility Mapper are running, the dispatcher UI needs a structured way to send richer incident data than the current 'POST /incidents' accepts. The 'POST /incidents/from-capture' endpoint is the integration point that makes the whole system coherent as a single data flow.

Tasks:

- [ ] 4.1 'POST /incidents/from-capture' in 'ambulance-cdss' — accepts 'CapturePayload' from shared contracts. Maps 'patientInfo.consciousness' → 'acvpu' for triage enrichment, maps 'incidentInfo.priority' → initial priority hint, maps 'patientInfo.activelyBleeding' → trauma protocol trigger hint. Calls the same 'create_incident' logic internally. Returns the standard 'create_incident' response plus '"capture_correlation_id"' echoing the 'dispatchId' from the payload. This allows the capture layer to correlate its own event log with the ambulance-cdss incident record.
- [ ] 4.2 Triage enrichment uses 'CapturePayload' vitals — 'gcs_score' derived from 'consciousness' field mapping ('"unconscious"' → GCS 3, '"responds to voice"' → GCS 9, '"confused"' → GCS 13, '"alert"' → GCS 15). 'activelyBleeding' adds TRAUMA modifier to enrichment request. These are imprecise but improve triage signal before vitals are formally recorded.
- [ ] 4.3 Facility routing uses enrichment output — when 'triage_enrichment.triage_level == "P1"' and top diagnosis includes cardiac/vascular keywords, add '"icu"' and '"cardiac"' to 'required_services' in facility routing. When 'triage_level == "P1"' and trauma keywords present, add '"surgery"' and '"blood_bank"'. This makes facility routing clinically aware, not just geographically nearest.
- [ ] 4.4 'POST /incidents/from-capture' integration test — verifies the full pipeline: capture payload in → incident created → triage enrichment fired → facility routing using enrichment-informed services filter.

Constraints:
- 'POST /incidents/from-capture' delegates to the same 'create_incident' logic. No copy-paste. The body mapping is the only addition.
- Consciousness-to-GCS mapping is documented with its limitations — it is an approximation, not a clinical assessment. This must be noted in the field name docstring and in the triage enrichment metadata.
- Facility service filter enhancement is optional (enrichment may not be resolved yet). The routing endpoint must not block waiting for it.

Acceptance:
- [ ] 'POST /incidents/from-capture' with a full 'CapturePayload' creates an incident, fires triage enrichment, and returns within 200ms
- [ ] 'GET /incidents/{id}/full' for that incident includes 'triage_enrichment' populated
- [ ] 'POST /incidents/{id}/route-facility' for a P1 cardiac incident returns facilities with '"cardiac"' in their services filter

### Phase 5 — Dispatcher UI hardening (~3 days)

Specific problems to fix, not general "UI improvements":

- [ ] 5.1 Offline detection banner — persistent top-of-page banner when API is unreachable. Not a header status badge — a full-width 'position: fixed' banner that cannot be missed. Colour: amber (degraded) not red (failed) — the dispatcher can still work from their last known state. Banner text: "API unreachable — working from last known state. New incidents cannot be created until connection is restored."
- [ ] 5.2 Answer submission loading state — all answer buttons disabled from click until next question renders. Add 'aria-busy="true"' on the question card during submission. No spinner graphic — just button 'opacity: 0.5' and 'pointer-events: none'. A spinner implies the dispatcher should wait and watch; the real instruction is "you clicked, it's processing."
- [ ] 5.3 Manual protocol selector — when 'protocol_matched: false' or 'requires_manual_verification: true', render a '<select>' populated from 'GET /field-protocols' and 'GET /protocols' active lists. Dispatcher selects, confirms, then incident is re-created with the correct protocol. This is not the same as the current "no match" state which just displays a warning with no action path.
- [ ] 5.4 Pre-arrival instruction timer — when terminal outcome renders, start a visible 'MM:SS' elapsed timer. Label: "Time since outcome reached: 00:00". Updates every second via 'setInterval'. Stops when dispatcher clicks "Confirm — instructions read to caller" (from the pre-arrival confirmation endpoint). This gives supervisors a time-on-instructions signal during quality review.
- [ ] 5.5 Triage enrichment display — when 'requires_manual_verification: true' or the incident has triage enrichment, show a collapsible "Clinical context" section below the question card. Contents: 'triage_level' badge (P1–P4 with colour coding), 'top_diagnosis' name, 'esi_level', 'shock_index' if available. Collapsed by default — a dispatcher running a P1 call does not need to read this, but a dispatcher with an ambiguous presentation does.
- [ ] 5.6 Keyboard navigation — all answer buttons reachable by Tab, activated by Enter or Space. Visible ':focus' outline on every interactive element. No mouse required for the full dispatch script flow.

Constraints:
- No new UI dependencies. Pure CSS and vanilla JS, same pattern as the current 'dispatcher-ui/'.
- Every change to the dispatcher UI must be tested with the existing 'tests/test_incident_lifecycle.py' integration tests. No new UI-layer test framework — the integration tests via ASGI transport are the right gate.
- Timer uses 'Date.now()' delta, not 'setInterval' counting. 'setInterval' drifts over long sessions. A timer that reads "00:31" after 30 seconds is wrong in a clinical context.

### Phase 6 — Field UI offline hardening (~3 days)

- [ ] 6.1 Write queue for offline actions — any 'POST'/'PATCH' that fails with a network error is stored in 'localStorage' as '{endpoint, method, body, queued_at}'. Queue key: 'ambulance_cdss_write_queue'. Queue drains in order (FIFO) when the next successful API call confirms connectivity. Queue is drained by a 'drain_write_queue()' function called at the start of every user action, not by a background 'setInterval' (avoids battery drain and complexity).
- [ ] 6.2 Optimistic UI for step marking — step buttons change to "Pending..." immediately on click, update to the confirmed status on API success, or revert with "Failed — queued" on API failure. The 'incident_field_log' append-only guarantee means a queued action, when eventually submitted, produces the correct outcome regardless of delay.
- [ ] 6.3 Vitals pre-population — when the vitals form is opened, if 'GET /incidents/{id}/full' has returned 'vitals_history' with at least one entry, pre-populate the form with the most recent values. A paramedic updating vitals is updating a delta, not starting from zero. Pre-population is labelled: "Pre-filled from last recording at [timestamp]".
- [ ] 6.4 Triage context on incident open — if the incident has 'triage_enrichment' populated, display it in a read-only "Dispatch triage" section on the checklist tab. 'triage_level', 'top_diagnosis', 'esi_level'. Not editable. This is context from dispatch, not a field assessment. Clearly labelled as such.
- [ ] 6.5 Offline mode indicator — same pattern as dispatcher UI. Full-width amber banner when API unreachable. Write queue count displayed: "3 actions pending sync."

Constraints:

### Code hygiene rules that apply across all phases

These are not guidelines — they are constraints. Any code that violates them is wrong and must be corrected before merging.

Python:
- Every function has a type-annotated signature. No 'def f(x)' — it's 'def f(x: str) -> Optional[int]'.
- No bare 'except:' or 'except Exception:' without a 'logger.warning' that names the exception variable. Swallowing exceptions silently is a patient safety risk — a failure that goes unlogged may go undetected.
- Every external HTTP call is wrapped in 'try/except' and returns 'None'/empty on failure. No external call can propagate an exception to an incident-creation or answer-submission path.
- 'asyncio.create_task()' tasks that fail must be caught in their own exception handler. An uncaught exception in a background task does not propagate to the caller — it disappears silently in Python's asyncio. Every 'create_task' call is paired with an '.add_done_callback' that logs exceptions.
- No '.env' files committed. '.env.example' always committed. The CI check runs 'git grep -r "DATABASE_URL=postgres" --include="*.env"' and fails if it finds credentials.
- 'ruff' passes with zero warnings on every file before it is committed. Config is identical across all three services.

Clinical content:
- Every 'guidance_note' in a protocol file that contains clinical instructions (dosing, technique, contraindication) must have a 'reviewed_by' field in a comment block above it. This is documentation of clinical accountability, not just code.
- Protocol files are never edited without the reachability validation test passing. The test suite is the gate for protocol changes, not a manual review.
- 'clinical_rules.yaml' changes require a comment explaining the clinical basis for the weight assigned. '# weight 0.8 — WHO BEC Course Module 4: airway obstruction is immediately life-threatening' not '# important term'.

Security:
- No PHI in logs, anywhere, in any service. This is enforced by a CI lint step that greps for 'incident_desc', 'description', 'chief_complaint' appearing in 'logger.*' calls.
- 'X-Admin-Key' is checked before any admin endpoint executes. The check is in a FastAPI dependency, not duplicated in each endpoint handler.
- CORS 'allow_origins=["*"]' is forbidden in any service's production configuration. 'validate_startup_config()' enforces this in every service that has a 'ENVIRONMENT=production' env var.
