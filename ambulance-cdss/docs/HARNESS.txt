
**What this system currently is:**

A backend API with two minimal static HTML consoles. That's it. No one can actually use it in a real scenario without understanding the API directly. The "dispatcher UI" and "field UI" are developer prototypes, not clinical tools.

**The three users and what they actually need:**

**Dispatcher / call-taker** (receives the emergency call)
- Needs: a console where they type/speak what the caller says, the system asks them structured questions (the locked script), they answer, and pre-arrival instructions are generated. They also need to assign a unit and route to a facility.
- Current state: `dispatcher-ui/` handles this flow. It works but is bare and fragile.

**Paramedic / field responder** (at the incident)
- Needs: a mobile-optimized interface (phone or tablet) where they follow a clinical checklist, record vitals, log medications, and see the triage context from dispatch. Offline-capable because they're in the field.
- Current state: `field-ui/` handles this but has no offline support and is not mobile-optimized.

**ER doctor / receiving clinician**
- Needs: to receive a structured handoff before the patient arrives — what happened, what was done, what the vitals trend was, what medications were given. They do NOT interact with the ambulance-cdss in real time. They receive the handoff document.
- Current state: `GET /incidents/{id}/handoff` generates this. But there is no screen for the ER doctor to view it. They'd need a URL with the handoff rendered as a readable page, or it gets printed/sent.

**What's missing that makes this feel incoherent:**

1. The dispatcher UI has no way to send the handoff to the receiving hospital. The document generates but goes nowhere.
2. The field UI is not usable on a phone in a moving ambulance.
3. There is no receiving-end UI — no screen an ER doctor opens to see the incoming patient's full clinical picture before they arrive.
4. The chronic-disease CDSS (`HIV-agent`) is completely separate. A doctor in an ER dealing with a patient who was already in the chronic-disease CDSS has no connection between the two systems.
5. The triage ranker enriches the incident but this enrichment is never surfaced to anyone in a meaningful, actionable way during the call.

**How it should work end-to-end in real life:**

```
Call comes in
    ↓
Dispatcher opens dispatcher console
Enters chief complaint → system matches protocol → locked script starts
Dispatcher answers questions guided by the caller
Terminal outcome fires → priority code + pre-arrival instructions
System assigns unit + routes to best facility
    ↓
Paramedic receives the incident on their phone (field UI)
Sees dispatch context + triage enrichment
Follows field protocol checklist
Records vitals (NEWS2/GCS computed automatically)
Logs medications
    ↓
Ambulance en route to hospital
System generates handoff document
Sends it to the receiving ER (currently: nothing does this)
    ↓
ER doctor opens receiving screen
Sees: incoming patient, priority, chief complaint, vitals trend,
medications given, triage level, ETA
Prepares the right team and equipment before arrival
```

**The three immediate gaps that make this real:**

**Gap 1 — No unit notification system.** When a unit is dispatched, nothing pushes the incident to the paramedic's device. The paramedic has to know their incident ID and manually open the field UI and type it in. In a real deployment, dispatch triggers a notification to the assigned unit's device.

**Gap 2 — No handoff delivery.** The handoff document is generated but never sent anywhere. The ER doctor has no screen and receives nothing automatically.

**Gap 3 — No receiving-end UI.** The ER doctor needs a simple read-only page: `GET /incidents/{id}/handoff` rendered as readable HTML with vitals trend visualized, not raw JSON.

**What to build next to make this coherent:**

In priority order:

1. **Receiving hospital UI** — a simple read-only HTML page at `/handoff/{incident_id}` served by the ambulance-cdss. Renders the handoff document as a clean, printable, clinical-grade summary. No login needed if accessed via a time-limited token. This is what the ER doctor opens when the ambulance calls ahead.

2. **Mobile-first field UI** — rework `field-ui/` for a 375px viewport, large touch targets, one action per screen rather than tabs with dense forms. A paramedic doing CPR cannot navigate tabs.

3. **Incident push to field unit** — when `POST /incidents/{id}/dispatch-unit` succeeds and `assigned_unit_id` is set, the field UI for that unit should automatically show the incident without the paramedic manually entering an ID. The simplest version: the dispatcher sends the paramedic a URL with the incident ID. No WebSocket needed yet.

4. **Handoff delivery** — when the incident reaches `TRANSPORTING` status and a facility is routed, automatically generate the handoff and make it available at a stable URL the dispatcher can send to the receiving hospital via any channel (SMS, WhatsApp, radio, phone).



Build all of the above exhaustively! No shortcuts! No emojis or stupid AI decorators/icons. This is a professional system!
