These three pass the cut — all grounded in what the codebase actually does, not speculative.

---
1. Shift handover report

What it is: 
'GET /dashboard/shift-handover?shift_start={iso}&shift_end={iso}' — returns a structured summary of all incidents within a time window: count by status and priority, list of incidents still active at shift end (with their current status, unit, overdue flag), and the three highest-priority incidents resolved during the shift with their timeline durations (dispatch-to-on-scene, on-scene-to-handoff). Plain-text rendering alongside the structured JSON, same pattern as the handoff summary.

Why it makes the cut: 
supervisors need this for every shift change. Right now the only option is 'GET /dashboard/stats' (counts only, no incident detail) and 'GET /dashboard/active-incidents' (active only, no historical context). A shift handover requires both in one call. All data already exists across 'incidents' + 'incident_dispatch_log' + the timestamp columns. Zero new tables, zero new models. One new repository function composing queries already written, one new endpoint, one new plain-text renderer.

What's not allowed: 
no LLM summary, no inference, no "suggested actions". Dates, counts, durations, statuses — nothing that isn't a direct database field or arithmetic on two timestamps.

2. Protocol answer correction window

What it is: 
for a fixed window after submitting an answer (configurable, default 60 seconds, stored in config as 'ANSWER_CORRECTION_WINDOW_SECONDS'), the dispatcher can call 'PATCH /incidents/{id}/answer/{log_id}' to correct the most recent answer. 
The correction: (a) writes a new 'incident_dispatch_log' row with 'is_backtrack=True' and the corrected answer, 
(b) marks the original row as superseded via a 'superseded_by' UUID column pointing at the new row, 
(c) re-runs 'submit_answer' on the locked runner against the corrected answer and returns the new 'current_question' or terminal outcome. Outside the window: 403 with '"correction_window_expired"'.

Why it makes the cut: 
'can_backtrack()' returns 'False' because unrestricted backtracking on a live locked script is dangerous. 
But a dispatcher mistyping "no" instead of "yes" within the first 60 seconds of an answer is a real, frequent, low-stakes error that the current system has no way to correct. 
The window constraint bounds the governance risk (you can't revise an answer from 10 minutes ago). 
The 'superseded_by' column preserves the full audit trail — no rows are deleted, the correction is itself a durable record. 
This is the right resolution of the 3.3b open decision, more nuanced than a blanket 'True'/'False'.

What's not allowed: 
no deletion of the original answer row. No backtracking to answers more than one step back. 
No correction after the window. 'superseded_by' is a new column on 'incident_dispatch_log' — one migration, one nullable UUID FK to the same table, not a new table.

3. Responder location update during active incident

What it is: 
'POST /incidents/{id}/unit-location' — accepts '{lat, lon, recorded_by, timestamp}' from the field unit at configurable intervals during an active call. 
Stores to a new lightweight 'incident_unit_location' table (incident_id, lat, lon, recorded_at — no more columns). 
'GET /incidents/{id}/unit-location/latest' returns the most recent location. 
The 'route_facility' endpoint ('POST /incidents/{id}/route-facility') currently uses the caller's location, which is where the patient is, not where the unit currently is — it should prefer the latest unit location as the search origin when one is available.

Why it makes the cut: 
the facility routing endpoint is currently routed from the patient's location (caller lat/lon at intake), which is correct initially. 
But by the time a paramedic on scene calls 'route-facility' to find the nearest receiving hospital, the unit is at the patient's location — which may be the same, but the routing should use the most current known location of the unit, not the intake coordinates from 15 minutes earlier. The unit-location table also feeds directly into a future dispatch-side map view with no additional work. 
The table is two columns of data per row, written once per interval, read once per routing call. Not over-engineered.

What's not allowed: 
no real-time streaming, no WebSocket, no geofencing, no route calculation. Just the raw coordinate stored and retrieved. The routing endpoint does a single-point nearest-facility query using whichever lat/lon is most current — no path computation, no ETA recalculation.
