What's broken, missing, or disconnected — grounded in the code:
---
1. 'dispatcher-ui/app.js' has duplicate 'processTranscript' function body
Lines ~280–320 define the suggestions rendering logic inside 'processTranscript', then the exact same block appears again after. This causes a JS syntax error — the second function body is orphaned code after the first function's closing brace. The UI will not load at all in a strict JS environment.
2. 'POST /incidents/{id}/transcript' does not exist in 'main.py'
The dispatcher UI calls 'PATCH /incidents/{id}/transcript' every 30 seconds to auto-save transcript. That endpoint is not in 'main.py'. Every auto-save call fails silently. The transcript is never persisted outside the in-memory locked-script log.
3. 'GET /incidents/{id}/stream' does not exist in 'main.py'
'receiving-ui/app.js' connects to 'GET /incidents/{id}/stream?token=...' for SSE. That endpoint does not exist. The receiving UI connects, fails, then exponential-backoff retries indefinitely. The ER doctor sees no live updates.
4. 'POST /auth/dispatcher-login' does not exist in 'main.py'
Both 'dispatcher-ui/app.js' and 'field-ui/app.js' call this on login. The endpoint does not exist. Neither UI can log in. Both are completely unusable.
5. 'POST /incidents/{id}/select-protocol' does not exist in 'main.py'
The manual protocol selector in the dispatcher UI calls this. No such endpoint. The dispatcher is stuck when protocol matching fails.
6. 'GET /incidents/{id}/unit-location/latest' does not exist in 'main.py'
Both dispatcher and receiving UIs poll this. Not implemented — only 'POST /incidents/{id}/unit-location' exists. The unit tracking map never updates.
7. 'POST /triage/extract-entities' does not exist in either 'ambulance-cdss/main.py' or 'triage-ranker/main.py'
The dispatcher speech listener calls this after each final transcript result. The endpoint does not exist. All entity extraction suggestions silently fail.
8. 'handoff_link.py' uses 'hmac.new' which does not exist — should be 'hmac.new'
'hmac.new' is not the correct Python API. It should be 'hmac.new(key, msg, digestmod)'. Actually Python's HMAC is 'hmac.new(key, msg, digestmod)' — but checking: 'hmac.new' IS the correct call in Python. The issue is: 'generate_handoff_token' creates a token but 'main.py' has 'GET /incidents/{id}/handoff-link' endpoint. Let me check — yes, 'handoff-link' endpoint calls 'generate_handoff_token' and returns a URL. But the receiving-ui injects '__INCIDENT_ID__' and '__TOKEN__' as literal placeholder strings — there is no server-side template rendering that replaces them. The receiving UI will always have 'INCIDENT_ID = "__INCIDENT_ID__"' and will immediately show the error screen.
9. 'receiving-ui/app.js' references 'data.eta_minutes' but 'HandoffSummary' has no 'eta_minutes' field
The handoff endpoint returns the 'HandoffSummary' dataclass which does not include 'eta_minutes'. The ETA countdown never starts.
10. 'receiving-ui' references 'el("sse-banner")', 'el("handoff-page")', 'el("error-screen")', 'el("clinical-alerts")', 'el("news2-alert")', 'el("news2-sparkline")' etc — none of these IDs exist in 'receiving-ui/index.html'
The receiving-ui HTML and JS are completely mismatched. Every 'el()' call in 'receiving-ui/app.js' returns 'null'. The entire receiving UI throws immediately on load.
11. 'dispatcher-ui/app.js' calls 'el("caller-lat")' and 'el("caller-lon")' in 'route-facility-btn' handler after the map was added — but the Leaflet map now writes these via 'placeIntakeMarker'. However the map uses 'reverseGeocode' which calls Nominatim directly from the browser, violating Nominatim's terms of service for bulk/production use without prior permission.
12. 'facility-mapper/app/geocoding.py' also calls Nominatim directly. Two separate services making unauthenticated Nominatim requests with the same User-Agent string will be rate-limited or blocked.
