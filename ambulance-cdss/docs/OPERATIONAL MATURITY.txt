Ambulance CDSS — Operational Maturity Implementation Plan

## Architecture Additions to the Original List

Additions:
   A: Dispatch protocol governance pipeline — the "approved_by: Dev Setup" values must be replaced with a real sign-off UI/workflow before any protocol can be considered production-
  authorised. This is the most critical correctness gap.
   B: Manual protocol application is a stub — the "Apply protocol" button in dispatcher-ui does nothing. This is a live dead code path.
   C: Field-ui never pings unit location — the backend model exists, the endpoint exists, the field-ui never calls it. Zero GPS data flows in normal use.
   D: Triage enrichment is not live — it's a background task but the UI never re-polls to show the result. A dispatcher who doesn't answer a question will never see it.
   E: Dashboard UI does not exist — no fleet/supervisor view at all.
   F: Field protocols are critically thin — 1 field protocol for 8 dispatch conditions. A paramedic attending a trauma, obstetric, or paediatric call has no checklist.
   G: PII purge job is wired to nothing —  purge_expired_incidents()  exists but no scheduler calls it.
   H: Handoff URL base is hardcoded to localhost — every production handoff link is broken.
   I: Audio transcription persistence — raw transcript should attach to the incident log for medico-legal chain.
   J: Authentication — no login, no session. Any browser opening the URL becomes a dispatcher. Dispatcher IDs are self-reported freeform text.

Removed from original list:
   Turn-by-turn navigation — out of scope for a CDSS. Surface the destination coordinate; let the device's native maps app take it from there.
   "AML/E911 integration as a first-party build" — replaced with: expose a well-documented webhook endpoint that an E911 system can POST to. We define the API surface, not the telecom
  stack.

  ### EPIC 1 — Listener: Real-Time Audio Capture & NLP Extraction (Dispatcher)
  Goal: Free the dispatcher's hands. The system must hear the caller, extract entities, and pre-populate the protocol — not wait for the dispatcher to type.

  #### 1.1 — Streaming Audio Capture in Dispatcher UI

  Tasks:
   Add a "Start listening" toggle button to the intake form in  dispatcher-ui/index.html
   Implement  Web Speech API  ( SpeechRecognition ) with interim and final result callbacks in  dispatcher-ui/app.js
   Implement graceful degradation: if browser does not support  SpeechRecognition , show a warning and fall back to manual entry — never block
   Mute the microphone when the dispatcher is speaking (push-to-listen or VAD threshold)

   Display a live rolling transcript panel (distinct from the Q&A transcript sidebar) in the intake screen during active listening
   Add a "Stop listening / manual mode" toggle
  Acceptance Criteria:
   Browser microphone activates on button click; permission prompt shown once
   Rolling transcript updates in near-real-time (< 500ms lag)
   Microphone icon shows active/inactive state visually
   If  SpeechRecognition  unsupported (Firefox without flag): clear "Manual entry only" notice appears; all other UI functional

  Constraints/Pitfalls:

    SpeechRecognition  is Chrome-only without polyfill — document this constraint for deployment; target browser must be Chrome or Edge
   No audio leaves the browser without explicit consent and transport encryption
   Interim results are noisy — only final results trigger entity extraction
   Background noise in a call centre is real — do not auto-submit anything; only suggest
  ──────
  #### 1.2 — NLP Entity Extraction (Backend)

  Tasks:

   Add  POST /triage/extract-entities  endpoint to  app/main.py
   Implement  app/external/nlp_extractor.py  as a new client (mirrors  triage_ranker.py  pattern — degrades gracefully to  None  if unconfigured)
   Entity types to extract: location (street address, landmark, area name), chief complaint (condition keywords), vitals (BP, HR, RR mentioned verbally), patient descriptors (age,
  gender, consciousness level)
   If  TRIAGE_RANKER_BASE_URL  is configured, pipe the transcript to the existing triage ranker first; add entity extraction as a second pass
   Add a  degraded_mode  path using local regex rules as fallback (no external service required): patterns for common Kenyan address formats, named landmarks, cardinal directions
   Return structured JSON:  { location_text, lat, lon, chief_complaint_suggestion, vitals, confidence }

  New Backend Route:

    POST /triage/extract-entities
    Body: { transcript: str, incident_id?: str }
    Response: { location_text, lat, lon, chief_complaint_suggestion, vitals: {...}, confidence: float, degraded_mode: bool }

  Acceptance Criteria:
   "Patient collapsed near Kenyatta Hospital, not breathing" → extracts  chief_complaint_suggestion: "cardiac arrest" ,  location_text: "Kenyatta Hospital"
   "chest pain, bp 180 over 90" → extracts vitals BP systolic=180 diastolic=90
   Endpoint returns within 2 seconds in degraded (local regex) mode
   Any failure returns  degraded_mode: true  with best-effort partial results — never a 500
  Pitfalls:
   Do not auto-apply extracted data — always surface as a suggestion with a confidence badge; human confirms
   Ambiguous location text (e.g. "near the school") must not be geocoded without confirmation
   Never write extracted data to the incident without dispatcher explicit confirmation click
  ──────

  Tasks:

  Cardiac Arrest — confirm?"

  Acceptance Criteria:

   Dismissed suggestions do not reappear unless new audio triggers them

  #### 1.5 — E911 / AML Webhook Receiver (Backend)
  Tasks:
   Add  POST /intake/e911-push  endpoint — accepts an E911 or AML location push payload
   Document the expected payload shape in a new  docs/E911_CONTRACT.md  file (this is the API surface we offer; the telecom partner implements against it)
   Extract  lat ,  lon ,  caller_number ,  accuracy_m  from the payload
   If an  incident_id  is present in the payload, update that incident's location; if absent, create a pre-populated intake record in  status: received  and return the  incident_id  so
  the dispatcher can open it
   Store  accuracy_m  on the incident (add column) so the dispatcher knows whether the pin is 5m or 500m accurate
   Emit a server-sent event on the incident's SSE stream (Epic 4) when location is updated
  New Backend Route:

    POST /intake/e911-push
    Body: { caller_number, lat, lon, accuracy_m, incident_id? }
    Response: { incident_id, created: bool }

  Pitfalls:

   Caller number is PII — must be subject to the same 30-day retention purge as  caller_location_*
   Accuracy matters clinically — a 500m radius in a city block is useless; display accuracy badge on the map pin

  ### EPIC 2 — Live Interactive Map (Dispatcher UI)

  Goal: Replace the lat/lon number inputs with a zoomable, clickable, geocoding-aware map. Location must be point-and-click, not numerical.

  #### 2.1 — Leaflet.js Map Integration (Dispatcher Intake)
  Tasks:
   Add Leaflet.js (CDN, no build step) to  dispatcher-ui/index.html
   Replace the  #caller-lat ,  #caller-lon  inputs and the  #caller-location-text  input with a map panel + a single address search bar
   Map renders at intake screen, centred on the organisation's operational area (configurable via  window.AMBULANCE_CDSS_MAP_CENTER  in  config.js )
   Clicking the map drops a pin and reverse-geocodes the coordinate to a human-readable address (use Nominatim/OpenStreetMap — no API key required; or Mapbox if a key is configured)
   Address search bar: typing an address forward-geocodes and pans the map + drops a pin
   Lat/lon values are captured as hidden state from the pin, not from direct user input
   When the E911 webhook fires (1.5), the pin auto-moves and pulses with an accuracy circle
   "Clear location" button removes the pin and clears the hidden lat/lon

  New  config.js  additions:
    window.AMBULANCE_CDSS_MAP_CENTER = [lat, lon]; // default: [-1.286389, 36.817223] (Nairobi)
    window.AMBULANCE_CDSS_MAP_ZOOM = 13;
    window.AMBULANCE_CDSS_GEOCODER = "nominatim"; // or "mapbox"
    window.AMBULANCE_CDSS_MAPBOX_TOKEN = ""; // only if mapbox

  Acceptance Criteria:

   Map loads within 2 seconds on a local network (tiles cached after first load)
   Clicking anywhere drops a pin within 100ms
   Reverse geocode result appears in the address display within 3 seconds
   Address search auto-completes with 3+ characters typed
   Pin coordinate is what gets submitted with the incident — not manually typed numbers
   Map degrades gracefully if geocoding service is offline: pin still works, address field shows raw coordinate string
   Map tiles load over HTTPS only

  Pitfalls:

   Nominatim has a usage policy — 1 request per second, must identify the application with a User-Agent header
   Do not request geolocation of the dispatcher's browser — the map is for the caller's location, not the workstation's
   No tiles should be cached by default in incognito/private mode — warn if tiles fail to load

  ### EPIC 4 — ER Dashboard: Live Incoming Patient View

  Goal: The ER team must see the patient's vitals trending live before the ambulance arrives — not a static printout.

  #### 4.1 — Server-Sent Events (SSE) Endpoint (Backend)

  Tasks:
   Add  GET /incidents/{id}/stream  SSE endpoint to  app/main.py
   Events streamed:  vitals_added ,  medication_added ,  status_changed ,  unit_location_updated ,  field_log_added
   Backend maintains an in-memory  asyncio.Queue  per active incident; each write endpoint notifies the queue
   SSE stream sends JSON-encoded events with  event:  type and  data:  payload
   SSE connection authenticated by the handoff token (same  ?token=  param as the handoff URL)
   Connection keepalive ping every 15 seconds
   Stream closes automatically when incident reaches  handoff_complete  or  closed

  New Backend Route:

    GET /incidents/{id}/stream?token=<handoff_token>
    Content-Type: text/event-stream

  Pitfalls:

   In-memory queues are lost on server restart — document this; SSE is ephemeral, not persistent
   Each open SSE connection holds an async generator — rate-limit to 10 concurrent SSE connections per incident
   Stale connections (client disconnected without clean close) must be garbage-collected
  ──────

  ### EPIC 6 — Authentication & Session Management

  Goal: "Dispatcher ID" cannot be a freeform text box. The system must know who is operating it.

  #### 6.1 — Dispatcher Login (Frontend)

  Tasks:

   Add a login gate page to  dispatcher-ui/index.html : username + PIN (4-digit minimum)
   On successful login, store  dispatcher_id  and a session token in  sessionStorage  (not localStorage — clears on tab close)
   All API calls include the session token in an  X-Dispatcher-ID  header
   Backend validates the  X-Dispatcher-ID  header on all write endpoints (not GET) — reject with 401 if absent or invalid in production mode
   Auto-logout after 8 hours of inactivity
   Dispatcher ID auto-populates the "Dispatcher ID" field in incident creation

  New Backend Behaviour:
   Add  POST /auth/dispatcher-login  endpoint: validates username+PIN against a configured list (initially:  DISPATCHER_CREDENTIALS  env var as JSON), returns a signed session token
  (HMAC, same pattern as  handoff_link.py )
   In  ENVIRONMENT=development , auth is bypassed — any dispatcher_id accepted
   In  ENVIRONMENT=production ,  validate_startup_config()  must assert  DISPATCHER_CREDENTIALS  is set

  Pitfalls:
   PINs must be stored hashed (bcrypt or SHA256+salt) — never plain text in env vars
   Session tokens must have an expiry; same HMAC+expiry pattern as handoff tokens
   A compromised session token exposes all incidents the dispatcher can reach — scope tokens to read-only vs. write
