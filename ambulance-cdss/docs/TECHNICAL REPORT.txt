TECHNICAL REPORT

System 1: CDSS Triage Ranker

Type: Clinical triage intelligence service
Port: 8100
Container: Podman/Docker

Core Function:
Three-stage NLP-driven triage system that processes emergency incident descriptions and outputs ranked clinical diagnoses with severity scoring.

Pipeline Architecture:
Stage 1: NLP Extraction (medspaCy)
  → Tokenization, entity extraction via TargetRules
  → ConText modifiers (negation, historical, severity)
  → Output: ExtractedKeyword[]

Stage 2: UMLS Resolution (async, cached)
  → Four-layer cache: L1 (TTLCache) → L2 (SQLite) → L3 (UMLS REST API) → L4 (fallback dict)
  → Resolves entities to CUI, semantic types, SNOMED-CT, ICD-10
  → Output: Enriched keywords with canonical medical codes

Stage 3: Clinical Ranking
  → Composite scoring: W_rule + W_semantic + W_modifier + W_scoring_system
  → Integrates GCS/ACVPU, Shock Index (HR/SBP), ESI scoring
  → Maps to severity levels (critical/high/acute/moderate/low)
  → Output: Ranked diagnoses with triage level (P1-P4) and ESI level (1-5)

Key Endpoints:
POST /triage — Main endpoint. Accepts incident description + optional clinical scores (GCS, SBP, HR, RR)
GET /health — Service health with UMLS reachability
GET /ready — K8s-style readiness probe
POST /admin/rules/reload — Hot-reload clinical rules without restart
DELETE /admin/cache — Purge cache layers

Request Schema:
TriageRequest(
    incident_desc: str,           # Required, 5-5000 chars
    gcs_score: int | None,        # Glasgow Coma Scale (3-15)
    acvpu: str | None,            # A/C/V/P/U consciousness level
    sbp: int | None,              # Systolic BP (30-300 mmHg)
    hr: int | None,               # Heart rate (20-300 bpm)
    include_umls_lookup: bool = True,
    semantic_type_filter: list[str] | None
)

Response Schema:
TriageResponse(
    diagnosis_ranking: DiagnosisRankItem[],  # Sorted by urgency
    historical_findings: DiagnosisRankItem[],
    keywords: ExtractedKeyword[],            # Raw extraction output
    triage_level: "P1"|"P2"|"P3"|"P4",
    esi_level: 1-5,
    degraded_mode: bool,
    metadata: {
        request_id, processing_times, cache_stats,
        shock_index, scoring_systems_used, inferred_risks
    }
)


DiagnosisRankItem structure:
{
    rank: int,
    canonical_name: str,
    umls_cui: str,
    snomed_code: str,
    icd10_code: str,
    severity_level: "critical"|"high"|"acute"|"moderate"|"low",
    esi_level: 1-5,
    score_breakdown: {
        w_rule, w_semantic, w_modifier, w_scoring_system, total
    },
    scoring_systems_applied: ["GCS_NUMERIC", "SHOCK_INDEX_CRITICAL"],
    modifier_classes: ["SEVERITY_CRITICAL", "ACTIVE"]
}


Degradation Strategy:
Stage 1 (NLP) and Stage 3 (Ranking) always run offline
Stage 2 degrades gracefully: API → Cache → Fallback → Skip
Service remains operational even with UMLS API down

Tech Stack:
FastAPI 0.115, Uvicorn, Pydantic v2
spaCy 3.8 en_core_web_md, medspaCy 1.x
SQLite cache (persistent volume), cachetools (in-memory)
UMLS Metathesaurus REST API (requires API key)
Python 3.11+

Deployment:
Container: Podman/Docker via compose.yaml
Resources: 512MB-768MB RAM, 0.5-2.0 vCPU
Volumes: .data/cache (SQLite), /rules (YAML configs, read-only)
Health checks: /ready probe with 90s startup grace period


System 2: Ambulance Protocols RAG API
This is what our system is built upon. 
The base of our system (by system I mean the general CDSS outside ambulance-CDSS directory) was forked from this repo at an earlier time when it didn't have some capabilities it does now.

Type: RAG-powered clinical protocol Q&A service
Port: Not specified (default 8000)
Container: Not containerized (standalone FastAPI app)

Core Function:
RAG system that answers emergency dispatch queries by retrieving relevant protocol sections from ambulance care documents (currently testing for KE) and generating actionable prehospital steps via LLM.

Pipeline Architecture:
Startup:
  → Ingest 2 PDF protocol documents into LanceDB vector store
  → Build text search retriever (k=5)
  → Initialize LLM client (OpenAI-compatible API)

Request Flow:
  1. Structure dispatch incident fields into rich query
  2. Semantic search LanceDB for top-5 protocol chunks
  3. Build prompt: system instructions + retrieved context + incident
  4. Call LLM via HTTP (OpenAI chat completions API)
  5. Parse JSON response {"protocols": [...]}
  6. Return ordered list of actionable steps with citations

Key Endpoints:
POST /em_protocols_rag — Main endpoint. Accepts structured dispatch incident
GET /health — Agent readiness status

Request Schema:
DispatchRequest(
    dispatchId: str,
    patientInfo: {
        ageGroup: str,
        approxAge: str,
        sex: str
    },
    incidentInfo: {
        description: str,          # Primary RAG input
        priority: str,
        consciousness: str,
        breathing: str,
        activelyBleeding: bool,
        medicalHistory: str
    }
)

Response Schema:
ProtocolResponse(
    protocols: list[str]  # Ordered actionable steps with citations
)

Example Response:
{
  "protocols": [
    "Ensure scene safety and don PPE [Ambulensi Dispatch Protocols][12]",
    "Assess airway, breathing, circulation (ABCs) [Ambulensi Prehospital Protocols][34]",
    "Control active bleeding with direct pressure [Ambulensi Prehospital Protocols][45]"
  ]
}

Data Sources:
Ambulensi Emergency Medical Dispatch Protocols (1).pdf
Ambulensi Prehospital Emergency Care Clinical Protocols (1).pdf
Vector DB: LanceDB (local filesystem storage)


Tech Stack:
FastAPI, Pydantic
LanceDB (vector store), custom embeddings backend
Direct HTTP LLM calls (no LangChain/pydantic-ai)
PDF ingestion via custom ingest.py module
Python 3.11+

Deployment:
Standalone Python app (not containerized in provided files)
Run from project directory: uvicorn api.main:app --reload
No explicit resource limits or health checks

Integration Considerations
When building a converged system consuming both services:

Call sequence: Triage Ranker first → Ambulance Protocols second
Triage ranker extracts structured clinical entities with severity
Use diagnosis_ranking[0] (top-ranked finding) + triage_level to enrich dispatch request to protocols API

Data flow:

Emergency Call Text
    ↓
POST /triage (Triage Ranker)
    ↓
{diagnosis_ranking, triage_level, esi_level, keywords}
    ↓
Build DispatchRequest using:
  - incident_desc → incidentInfo.description
  - diagnosis_ranking[0].canonical_name → enhance description
  - triage_level → incidentInfo.priority
  - Extract vital signs from keywords → consciousness, breathing
    ↓
POST /em_protocols_rag (Ambulance Protocols)
    ↓
{protocols: [...]} — Actionable prehospital steps


Shared data mapping:
| Triage Ranker Output | Ambulance API Input | 
|---------------------|-------------------| 
| incident_desc (original) | incidentInfo.description | 
| diagnosis_ranking[0].canonical_name | Append to description | 
| triage_level ("P1"-"P4") | incidentInfo.priority | 
| esi_level (1-5) | Map to priority descriptor | 
| keywords with category VITAL_STATUS | incidentInfo.consciousness | 
| keywords with sbp, hr if provided | Structured vitals (not in current schema) | 
| diagnosis_ranking[].modifier_classes contains SEVERITY_CRITICAL + category CLINICAL_SIGN matches "bleeding" | incidentInfo.activelyBleeding = true |

Error handling:
Triage ranker degrades gracefully (operates offline if UMLS fails)
Ambulance API fails hard if LLM unavailable (503 during startup, 500 on LLM errors)
No retry logic in ambulance API — implement at integration layer

Latency expectations:
Triage ranker: 200ms-5s (depending on UMLS cache hits)
Ambulance API: 2-10s (LLM generation + retrieval)
Combined: 2-15s total for end-to-end dispatch support

PHI safety:
Triage ranker: PHI-safe logging (no patient text in logs)
Ambulance API: No explicit PHI controls — patient data sent to external LLM
Integration layer must handle PHI compliance (anonymization, consent, data residency)

Authentication:
Triage ranker: Admin endpoints require Bearer token (ADMIN_API_KEY)
Ambulance API: No authentication (assumes internal network)
Add API gateway auth layer for production

Caching strategy:
Triage ranker caches UMLS lookups (persistent SQLite + in-memory TTL)
Ambulance API has no response caching — every call hits LLM
Consider caching protocol responses by incident signature for cost optimization
The triage ranker provides structured clinical intelligence (diagnosis, severity, standardized codes), while the ambulance API translates that into human-readable, protocol-grounded action steps. 
Use triage ranker output to construct enriched queries for the ambulance API.

Type: Vanilla JS module (bundled to IIFE)

3. Facility Mapper API (api.py, port 8001)
Type: Geospatial hospital matching service
Stack: FastAPI, geopy, scikit-learn (BallTree), pandas

Data Source:
health_facility_data_feb11.json — Kenya/UK Master Health Facility List (Feb 2024)
10,000+ licensed facilities with GPS coordinates

Core Algorithm:
1. Geocode incident location → (lat, lon) via Nominatim
2. Load facility database (cached for 1 hour)
3. BallTree KNN search on [lat, lon] coordinates
4. Filter to Level 4+ facilities only
5. Calculate distance (km) and ETA (minutes at ambulance speed)
6. Sort by distance, return top 3

Level Classification:
Level 4+: Comprehensive care (surgery, blood bank, ICU)
Level 3: 50% services (basic ER, limited surgery)
Level 2: Basic services (no advanced trauma care)

Endpoints:
| Endpoint | Method | Query Params | Returns | 
|----------|--------|--------------|---------| 
| /health | GET | — | Service status + facility count | 
| /facility_mapper | GET | location, radius_km, max_results | Top 3 Level 4+ hospitals | 
| /nearest_hospitals | GET/POST | Same as above | Same response (legacy alias) | 
| /predict | POST | scene_location, patient_info | Triage + UMLS analysis |

Response Schema:
{
  "scene_location": "Kiambu" or "London",
  "top_hospitals": [
    {
      "rank": 1,
      "facility_name": "Kiambu/London Level 5 Hospital",
      "county": "Kiambu"  or "London",
      "level": "Level 4+ (All services)",
      "phone": "+254....", or "+4......"
      "distance_km": 2.3,
      "eta_minutes": 4.1
    }
  ],
  "total_found": 3
}

Speed Calculation:
Ambulance: 60 km/h (realistic urban/rural mix)
Formula: ETA = (distance_km / 60) * 60 minutes

UMLS Integration (Optional):
If UMLS_API_KEY provided → enriches symptoms with medical codes
/predict endpoint performs triage scoring + UMLS concept mapping
Note: Current implementation uses basic keyword triage, not the advanced triage ranker service
