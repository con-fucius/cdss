## Critical observations before the plan

What's true from the code that changes enrichment scope:

1. 'models.py' has no patient encounter, vitals, labs, medication, or diagnosis tables. The enrichment 'Phase A' is a greenfield schema addition — the only existing patient model is 'PatientRef' (hash-only) and 'LongTermMemory' (free-text facts). The 'PatientContext' Pydantic model is transient — no persistence. This makes Phase A the real prerequisite for everything.

2. 'api.py''s 'PatientContext' is a shallow dict. It has 'medications: List[str]' (already wired to drug-interaction checks), 'active_conditions: List[str]', 'clinical_params: Dict', 'comorbidity'. These are the fields scoring and pathway engines must accept as input — they already exist in-flight but are never persisted.

3. The 'external/clients.py' is implemented and wired. The 'app/external/' module exists as the right home for future external clients per the enrichment doc's pattern.

4. 'evidence.py' has 'format_evidence_triples' already returning structured data — Phase E's document generator can pull from this directly.

5. No 'scoring.py', 'ddx.py', 'pathways/', or 'documents.py' exist. All enrichment phases are net-new modules.

6. The enrichment's alert governance (Phase G) is described as a new table + override tracking. The existing 'AuditLog' table's 'event_type' field is already used for 'CLINICAL_SCORE', 'DRUG_INTERACTION_CHECK' etc. — alert override records fit cleanly into 'AuditLog' without a new table. This avoids schema sprawl.

7. The enrichment doc's CDS Hooks (Phase F) endpoint contracts are well-specified and map cleanly onto the existing FastAPI app. The 'RateLimitMiddleware' and 'MetricsMiddleware' apply automatically.

Pushbacks on enrichment.txt:

- Phase A's SQL schema uses 'patient_ref TEXT NOT NULL' — but the existing codebase hashes patient refs via 'patient_ref_from_context()' and stores only 'patient_hash' in 'PatientRef' and 'LongTermMemory'. The enrichment schema must use 'patient_ref TEXT NOT NULL' as a hashed value following the same pattern — not raw identifiers. Don't deviate from the existing de-identification approach.
- Phase E's SBAR includes 'Differential rank: [if DDx was run]' — this creates an ordering dependency (E requires C). Plan must enforce this.
- Phase G says "override tracking schema should be added with Phase B." The plan enforces this: B ships with G1+G2 (schema), G3 (admin endpoint) is its own step after sufficient override data exists to be useful.
- The enrichment doc recommends starting NEWS2 and eGFR first in Phase B. Correct — they are the highest-frequency scores in this disease scope and have the smallest input surface.
- Phase D's pathway runner references 'evidence_graph edges' directly — 'query_evidence_graph_db' in 'repositories.py' already filters by 'relation_type'. The pathway runner can query by specific relation types (e.g. 'drug_requires_monitoring', 'contraindicated_with') without new repo methods.

### B1 — Scoring engine module

Problem
No 'scoring.py' exists. Scorers described in enrichment.txt are all deterministic — no LLM, no retrieval — but are not implemented.

Tasks
- [ ] New module 'app/scoring.py' with class 'ClinicalScorer'.
- [ ] Implement in this exact order (highest clinical impact first):

  NEWS2 ('news2(vitals: dict) -> dict')
  - Inputs: 'rr', 'spo2', 'spo2_scale' (1 or 2), 'supplemental_o2' (bool), 'bp_systolic', 'heart_rate', 'consciousness' ("A"/"V"/"P"/"U"), 'temperature'.
  - Each parameter scores 0-3 per NHS NEWS2 specification (2017). Sum is total score.
  - Returns: '{score: int, risk_level: str, component_scores: dict, escalation_required: bool, trigger: str, source_guideline: "NHS NEWS2 2017"}'.

  eGFR / CKD staging ('egfr_ckd_stage(creatinine: float, age: int, sex: str) -> dict')
  - CKD-EPI 2021 equation (race-free). Returns '{egfr: float, ckd_stage: str, drug_implications: List[str], source_guideline: "CKD-EPI 2021"}'.
  - 'drug_implications' for this disease scope: TDF dose adjustment if eGFR < 50; metformin contraindicated if eGFR < 30; ACE inhibitor caution if eGFR < 45.

  WHO HIV staging ('who_hiv_stage(clinical_features: List[str], cd4: Optional[float]) -> dict')
  - Map clinical features against WHO Stage 1-4 criteria (Kenya ARV Guidelines 2022, Annex). Return matching stage and unmet criteria for the next stage.
  - Returns: '{stage: int, criteria_met: List[str], criteria_not_met_for_next_stage: List[str], recommended_actions: List[str], source_guideline: "WHO/Kenya ARV Guidelines 2022"}'.

  Child-Pugh ('child_pugh(labs: dict, clinical: dict) -> dict')
  - Inputs: 'bilirubin' (µmol/L), 'albumin' (g/L), 'inr', 'ascites' ("none"/"mild"/"moderate-severe"), 'encephalopathy' (0-4 grade).
  - Returns: '{score: int, class: "A"|"B"|"C", drug_dosing_implications: List[str], source_guideline: "Child-Pugh (Pugh 1973, updated)"}'.

  Malaria severity ('malaria_severity(vitals: dict, labs: dict, clinical: dict) -> dict')
  - WHO severe malaria criteria (Kenya Malaria Guidelines 2023): GCS < 11, RR > 30, BP systolic < 90, Hb < 7 g/dL, blood glucose < 2.2 mmol/L, parasitaemia > 2%, creatinine > 265 µmol/L, urine output < 0.5 mL/kg/hr, jaundice.
  - Returns: '{is_severe: bool, criteria_met: List[str], criteria_count: int, recommended_management: str, source_guideline: "Kenya Malaria Guidelines 2023"}'.

  HbA1c / DM assessment ('diabetes_risk_hba1c(hba1c: Optional[float], fpg: Optional[float]) -> dict')
  - WHO/Kenya DM Guidelines V15 2024 thresholds. Returns '{diagnosis: str, target_met: bool, intensification_indicated: bool, source_guideline: "Kenya DM Guidelines V15 2024"}'.

## Directive addendum (applies on top of 'directive.txt')

- Do not modify 'directive.txt' rules — they apply here unchanged.
- Phase A adds new Alembic migrations (0007, 0008, 0009). After each: run 'alembic upgrade head', verify 'alembic heads' is single-head before proceeding.

- 'ClinicalScorer' methods are synchronous, pure Python. Do not make them async, do not add LLM calls, do not import from 'api.py' (circular).
- 'DDxEngine', 'PathwayRunner', 'ClinicalDocumentGenerator' may call async functions but must not import directly from 'api.py'. Any shared helpers (e.g. '_compute_patient_scores') must be moved to their own module if needed by multiple callers — do not create circular imports.
- Frontend 'EventSource' does not support POST. DDx and pathway streaming use the same 'fetch' + 'ReadableStream' pattern already in 'useChat.js'. Do not introduce 'EventSource' for new streaming endpoints.
- 'alert_overrides.override_reason' is a Postgres CHECK constraint. Do not enforce it only in Python — the DB must reject invalid values independently.
- Pathways with 'steps=[]' must never crash the runner. They must yield a structured '{"type": "warning"}' event. This is the initial state for 9 of 12 pathways.
- Document generator uses the cheapest available model — verify which model 'memory.py''s 'distill_session_candidates' uses and replicate that pattern exactly.
- Every new endpoint: 'AuditLog' write. No exceptions.
