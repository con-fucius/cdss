Non-negotiable rules
1. Read before writing. For every file listed in a phase, 'view' the current content immediately before editing. Do not edit based on earlier context — files change between phases.
2. No new frameworks, libraries, or services unless explicitly named in a task (e.g. RxNorm client in Phase 5). If you think you need one, stop and ask.
3. No new abstractions beyond what's specified. Do not introduce base classes, plugin systems, generic "managers," config layers, or interfaces "for future flexibility." If a task says "one function," write one function.
4. Feature flags default to 'false'/off wherever specified. Verify the default produces zero behavior change before touching the enabled path.
5. Copy the pattern used by '_query_evidence_context' or '_shadow_umls_telemetry' — don't invent a new error-handling style.
6. Preserve existing behavior for the unaffected path. Every phase has a "diseases/queries without X behave identically" checklist item. Treat this as a hard requirement — write or run a quick before/after comparison, don't assume.
7. One phase at a time, in order. Do not start Phase N+1 until Phase N's acceptance checklist is fully checked. Do not reorder for convenience.
8. No silent scope expansion. If implementing a task reveals a related "obvious improvement" (e.g. "while I'm here, let me also refactor X"), do not do it. Note it separately and ask.
9. Match existing code style exactly: naming conventions, docstring format, logging style ('logger.warning("...: %s", exc)'), async patterns, type hints. Look at the nearest analogous function in the same file before writing new code.
10. Windows/PowerShell environment. Any commands, scripts, or paths you produce must work in PowerShell, not bash, unless the existing script is already '.sh'/bash-based (don't convert existing scripts' shells).

Anti-patterns to actively avoid
- Under-engineering: don't skip the dedup/fallback/timeout logic specified just to "get it working" — these are safety-relevant (clinical context).
- Over-engineering: don't add config options, env vars, or parameters not specified. Don't make things "pluggable" or "extensible" beyond the literal task.
- Dead-code accumulation: if a task says replace/extract logic from function A into function B, A must actually call B afterward — no duplicated logic left behind.
- False degradation states: per Phase 5.2 — never let an "external service unavailable" case produce output that looks like "checked, nothing found." These are different facts and must be represented differently.
- Silent fallback that hides bugs: graceful degradation must log at 'warning' level with enough detail to debug. Don't swallow exceptions silently.
- Frontend storage rules: no 'localStorage'/'sessionStorage' in any new artifact-style code; for the existing 'useChat.js', do not touch the storage migration shims unless a phase explicitly says to.
